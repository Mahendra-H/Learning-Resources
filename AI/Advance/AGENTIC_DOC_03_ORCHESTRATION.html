Why Orchestration Became a Discipline

The problem that emerged when teams discovered: great agents + no coordination = chaos.
🎬 The Origin — The Coordination Crisis of 2023

By mid-2023, teams had figured out how to build capable individual agents. The next problem hit immediately: connecting them was a disaster. Agent A would produce output in a format Agent B couldn't parse. Agent B would wait indefinitely for Agent C, which had silently failed. Agent D would make API calls that overlapped with Agent A's calls, producing duplicate results. What worked perfectly in isolation produced garbage when connected.

The engineers who got multi-agent systems working weren't necessarily using better agents — they were applying better coordination patterns. These patterns were rediscovered independently by dozens of teams before being formalized. The core insight: there are really only four ways agents can be structured relative to each other — sequential, parallel, hierarchical, and event-driven. Everything else is a variation.

LangChain's callback system and chain composition were the first codified attempt. CrewAI's Process types were the next evolution. Today, understanding these four patterns is non-negotiable for anyone building production agentic systems. It's the difference between a system that works and one that occasionally produces good output by accident.

🎬 The Orchestra Analogy

A solo violinist is impressive. A 60-piece orchestra playing without a conductor is noise. Each musician is individually talented — the problem is coordination. The conductor (orchestration layer) doesn't play an instrument. Their job is: who plays when, how loud, at what tempo, and how to recover when the cellos come in a beat late. Your orchestration patterns are the conductor's score. LangChain provides the instruments. CrewAI provides the musicians. This module teaches you how to write the score.

✅ Orchestrated — clean patterns with CrewAI
from crewai import Agent, Task, Crew, Process
from crewai_tools import SerperDevTool

search = SerperDevTool()

researcher = Agent(role="Researcher", goal="Research {topic}",
  backstory="Expert analyst.", tools=[search])
analyst    = Agent(role="Analyst",    goal="Extract insights",
  backstory="Turns data into decisions.")
writer     = Agent(role="Writer",     goal="Write clear reports",
  backstory="Executive comms specialist.")

# Explicit dependencies — no ambiguity
t1 = Task(description="Research {topic}",
  expected_output="Bullet list of key facts with sources", agent=researcher)
t2 = Task(description="Analyse the research",
  expected_output="Top 3 insights + risk assessment",
  agent=analyst, context=[t1])  # ← explicit dependency
t3 = Task(description="Write executive report",
  expected_output="2-page report: Title, Findings, Actions, Risks",
  agent=writer, context=[t1, t2], output_file="report.md")

# Process defines the coordination pattern
crew = Crew(agents=[researcher, analyst, writer],
  tasks=[t1, t2, t3], process=Process.sequential, verbose=2)
result = crew.kickoff(inputs={"topic": "AI market 2025"})

The 4 Orchestration Patterns

Every multi-agent system is a variation of one of these four. Know them cold.
📐 Diagram 1 — The 4 Orchestration Patterns: When to Use Each
🧠 The Decision Rule

Ask these questions in order: (1) Do tasks depend on each other in a fixed sequence? → Sequential. (2) Can some tasks run independently at the same time? → Parallel. (3) Do tasks change dynamically based on earlier results, or does a manager need to review? → Hierarchical. (4) Does the workflow start from external triggers rather than a single kickoff? → Event-driven. Most CrewAI use cases are Sequential or Hierarchical. Parallel and Event-driven require more custom infrastructure.

Building a Custom Tool — The @tool Decorator

LangChain's @tool decorator turns any Python function into something a CrewAI agent can call. The docstring is the tool's description — write it clearly because the LLM reads it to decide when to use the tool.

python · custom tools for CrewAI agents
from langchain.tools import tool
from crewai import Agent
import requests

# ── Custom Tool 1: Company funding lookup ────────────────────
@tool
def get_funding_data(company_name: str) -> str:
  """Looks up funding history for a company by name.
  Use this when you need to find investment rounds, total
  funding amount, lead investors, or last funding date.
  Input: company name as a string."""
  # Replace with your actual data source / API
  url = f"https://api.crunchbase.com/entities/{company_name}"
  try:
    data = requests.get(url, headers={"X-cb-user-key": "your-key"}).json()
    return f"Funding: {data.get('total_funding', 'Unknown')} | Last round: {data.get('last_round', 'Unknown')}"
  except:
    return "Could not retrieve funding data for this company."

# ── Custom Tool 2: Internal knowledge base ───────────────────
@tool
def search_company_kb(query: str) -> str:
  """Searches our internal company knowledge base.
  Use this for questions about our products, pricing,
  sales playbooks, and customer case studies.
  Do NOT use for public/internet information."""
  # Could query a LanceDB/Pinecone vector store
  from vectorstore import search_kb
  results = search_kb(query, top_k=3)
  return "\n".join([r["text"] for r in results])

# ── Use both tools in a CrewAI agent ─────────────────────────
sales_agent = Agent(
  role="Sales Intelligence Specialist",
  goal="Find the right approach to sell to each prospect",
  backstory="Expert at combining public research with internal playbooks.",
  tools=[get_funding_data, search_company_kb],   # ← direct reference
  verbose=True
)
# The agent will decide: "I need funding data? Call get_funding_data."
# "I need our sales playbook? Call search_company_kb."
# "I need web info? I don't have a search tool — I'll say so."
💡 Write Tool Docstrings Like You're Briefing a New Hire

The agent reads the docstring to decide whether to use a tool. """Gets funding data""" is not helpful. """Looks up funding history for a company. Use this when you need investment rounds, total funding, lead investors, or last funding date.""" tells the LLM exactly when to use it and what it returns. The quality of your tool descriptions directly impacts the quality of tool selection.

Mapping Business Processes to Agent Crews

Every business workflow has an agentic equivalent. Here's how to translate one to the other.

The most common mistake when designing a crew: building from tools up instead of from the business process down. Start with the workflow you want to automate, draw the steps, identify the handoffs, then map each step to an agent.

📐 Diagram 3 — Business Process to Agent Crew: The Translation Map
🧠 The Mapping Rules

Business process step → Agent. Person's job title → Agent role. What they're responsible for → Agent goal + backstory. Their tools → Agent tools list. What they hand off → Task expected_output. What they receive from the previous person → context=[previous_task]. The process flow (serial vs review-based) → Sequential vs Hierarchical.

Error Handling in Orchestrated Systems

What happens when Agent 3 fails in a 5-agent pipeline? You need a plan before it happens.

This is the section most tutorials skip. Production orchestrated systems will fail — tools return errors, APIs time out, agents produce unparseable output. Your crew design must account for this from day one.

python · production-grade crew with error handling
from crewai import Agent, Task, Crew, Process
from langchain.tools import tool
from typing import Optional
import logging

logger = logging.getLogger(__name__)

# ── 1. Tools that fail gracefully ────────────────────────────
@tool
def safe_web_search(query: str) -> str:
  """Search the web for information. Returns results or error message."""
  try:
    from crewai_tools import SerperDevTool
    result = SerperDevTool().run(query)
    return result
  except Exception as e:
    logger.error(f"Search failed: {e}")
    return f"Search unavailable. Reason: {str(e)}. Please use available knowledge."
    # Agent receives this message and adapts — doesn't crash

# ── 2. Crew with retry and fallback ──────────────────────────
def run_crew_safely(crew: Crew, inputs: dict,
                    max_retries: int = 2) -> Optional[str]:
  """Run a crew with retry logic and structured error reporting."""
  for attempt in range(max_retries + 1):
    try:
      result = crew.kickoff(inputs=inputs)
      logger.info(f"Crew completed successfully on attempt {attempt+1}")
      return result
    except Exception as e:
      logger.warning(f"Crew attempt {attempt+1} failed: {e}")
      if attempt == max_retries:
        logger.error("All retry attempts exhausted")
        return None   # caller handles None gracefully
  return None

# ── 3. QA Agent as the last line of defence ──────────────────
qa_agent = Agent(
  role="Quality Assurance Specialist",
  goal="Verify outputs are complete, accurate, and usable",
  backstory="""You are the last check before output reaches the user.
  Your job: if the output is incomplete, incorrect, or contains
  error messages from failed tools, return 'INCOMPLETE: [reason]'
  instead of passing garbage forward.""",
  verbose=True
)

qa_task = Task(
  description="Review the output from all previous agents. If complete and accurate, return it formatted. If incomplete or containing errors, return 'INCOMPLETE: [specific reason]'.",
  expected_output="Either the completed, formatted output OR 'INCOMPLETE: [reason]'",
  agent=qa_agent,
  context=[...all_previous_tasks...]
)
# The QA agent is your canary — it catches
# silent failures before they reach the user.
⚡ The Three Rules of Production Error Handling

(1) Tools should never raise exceptions to the agent — they should return error messages. An agent that receives "Search unavailable" can adapt. An agent that crashes mid-task takes the whole crew down. (2) Always have a QA agent as the final task. It's cheap insurance — one extra LLM call that catches silent failures. (3) Wrap kickoff() in try/except with retry logic. Transient API failures are common. Two retries with exponential backoff handles 95% of them.

Real-World Scenario — Customer Support Automation

500 tickets a day, 2-hour manual triage, 4 agents, 8 minutes end-to-end.
🎧
E-Commerce Platform · 500 tickets/day · 12-person support team
Hierarchical 4-agent crew: Triage → Specialist → Resolver → QA
🔴 The Problem

The support team spent 2 hours daily just triaging incoming tickets — reading, classifying, and routing to the right specialist. 30% of tickets were misrouted, causing frustrating back-and-forth. Response quality was inconsistent between agents. CSAT scores were declining.

🟢 The CrewAI Solution

Hierarchical crew: Triage Agent classifies tickets (billing/tech/general) → Specialist Agent handles the domain → Resolver Agent writes the response → QA Agent verifies quality and flags escalations. Human agents only see pre-triaged, pre-researched tickets with a draft response ready to approve.

Result: Triage time: 2 hours → 8 minutes. Misrouting: 30% → 4%. CSAT: +22 points. Human agents now handle escalations and relationship-building, not classification and research. The crew processes 500 tickets while the team handles 40 high-touch cases.
python · support_automation_crew.py — hierarchical process
from crewai import Agent, Task, Crew, Process
from crewai_tools import SerperDevTool
from langchain.tools import tool
from langchain_openai import ChatOpenAI

llm    = ChatOpenAI(model="gpt-4o-mini", temperature=0.2)
search = SerperDevTool()

@tool
def lookup_order(order_id: str) -> str:
  """Look up order status, shipment info, and history by order ID."""
  return f"Order {order_id}: Shipped 2024-01-15, expected delivery 2024-01-18, carrier: FedEx #123456"

@tool
def get_refund_policy(category: str) -> str:
  """Retrieve refund policy for a product category."""
  policies = {
    "electronics": "30-day returns, must be in original packaging",
    "clothing":    "60-day returns, no tags required",
    "default":     "30-day return policy, contact support for details"
  }
  return policies.get(category, policies["default"])

# ── Agents ───────────────────────────────────────────────────
triage_agent = Agent(
  role="Support Triage Specialist",
  goal="Classify incoming tickets accurately and extract key information",
  backstory="5 years routing 500+ tickets/day. You spot billing vs tech vs general issues instantly and always extract the order ID, product, and urgency level.",
  llm=llm, verbose=True
)

specialist_agent = Agent(
  role="Customer Support Specialist",
  goal="Research the customer's issue and find the accurate resolution",
  backstory="Deep knowledge of our product catalog, policies, and common issues. You use order lookup and policy tools to find the specific answer, not generic responses.",
  tools=[lookup_order, get_refund_policy, search],
  llm=llm, verbose=True
)

resolver_agent = Agent(
  role="Response Writer",
  goal="Write clear, empathetic customer responses that fully resolve the issue",
  backstory="Expert at de-escalating frustrated customers. You always: acknowledge the problem first, provide the specific answer, give a clear next step, and end with genuine warmth. Never generic, never robotic.",
  llm=llm, verbose=True
)

qa_agent = Agent(
  role="Quality Assurance Agent",
  goal="Verify responses are complete, accurate, and customer-appropriate",
  backstory="Last line of defence. If a response is incomplete, contains wrong info, or is too long, you either fix it or flag 'ESCALATE: [reason]' for human review.",
  llm=llm, verbose=True
)

# ── Tasks ────────────────────────────────────────────────────
triage_task = Task(
  description="Analyse this ticket: {ticket_text}. Extract: category (billing/shipping/technical/general), urgency (high/medium/low), order ID if mentioned, and one-sentence summary of the issue.",
  expected_output="JSON: {category, urgency, order_id, summary}",
  agent=triage_agent
)

research_task = Task(
  description="Based on the triage result, research the specific resolution. Use tools to look up order status or relevant policies. Find the exact answer, not a generic one.",
  expected_output="Factual answer to the customer's issue with specific data (tracking number, refund amount, policy section, etc.)",
  agent=specialist_agent,
  context=[triage_task]
)

response_task = Task(
  description="Write the customer response using the triage context and research findings. Empathetic opening, specific resolution, clear next step. Max 120 words.",
  expected_output="Customer-ready email response, max 120 words, with Subject: line",
  agent=resolver_agent,
  context=[triage_task, research_task]
)

qa_task = Task(
  description="Review the draft response. Check: is the issue fully addressed? Is the tone appropriate? Is it factually accurate? Return the approved response or 'ESCALATE: [reason]'.",
  expected_output="Final approved response OR 'ESCALATE: [specific reason]'",
  agent=qa_agent,
  context=[triage_task, research_task, response_task]
)

# ── Hierarchical crew — manager reviews at each step ─────────
crew = Crew(
  agents=[triage_agent, specialist_agent, resolver_agent, qa_agent],
  tasks=[triage_task, research_task, response_task, qa_task],
  process=Process.sequential,   # sequential is fine here — clear linear flow
  verbose=2
)

result = crew.kickoff(inputs={
  "ticket_text": "Hi, I ordered a jacket (Order #A12345) 10 days ago and it still hasn't arrived. The tracking number says it was delivered but I never received it. This is urgent as I need it for an event this weekend."
})
print(result)

Myth vs Reality

Three orchestration myths that cost teams days of debugging.
MYTH"Parallel agents are always faster. Just run everything simultaneously and merge at the end."
REALITYParallel execution only helps when tasks are truly independent. If Agent B's task relies on context from Agent A's output — even subtly — running them in parallel produces stale inputs and silent quality degradation. Before making tasks parallel, ask: "Could Agent B produce equally good output if it had zero knowledge of what Agent A found?" If no, they must be sequential. Force parallelism on dependent tasks and you'll spend days debugging why the final output keeps being inconsistent. The classic mistake: parallelising research tasks where each agent's findings should inform the others.
MYTH"Orchestration is about making agents talk to each other. Just set up a communication channel between them."
REALITYOrchestration is about coordination and control flow — not communication. Agents don't "talk" to each other directly in CrewAI. They produce structured outputs, which are stored as task results, which downstream agents receive as context. The orchestration layer (the Process type and task dependency graph) defines who works when, what they receive, and in what order. Thinking of it as "communication" leads people to build complex message-passing systems when a clean context=[] dependency graph is all they needed. Simpler coordination beats complex communication every time.
MYTH"You need sophisticated orchestration for any serious multi-agent system. Simple sequential is just for demos."
REALITYThe majority of production multi-agent systems in the wild use straightforward sequential orchestration with clean task dependencies. Sequential is predictable, debuggable, and cheap. The teams running the most reliable agentic systems in production aren't using exotic patterns — they're using well-defined sequential pipelines with good error handling and clear expected_output specifications. Reach for complex orchestration patterns when you have a specific, demonstrated need. Adding complexity to "seem sophisticated" is how you build systems that impress in demos and fail in production.

FAQ — The Questions Everyone Has

Real questions. Straight answers. No fluff.
Q1When does orchestration actually help vs add unnecessary complexity and cost?›
The break-even point: orchestration pays off when the coordination overhead (extra LLM calls, latency, cost) is smaller than the quality improvement from specialisation and structured handoffs. Rule of thumb: if a single agent with all the tools would produce output that's "good enough," use a single agent. If genuine specialisation — a dedicated researcher finding things a generalist misses, a dedicated QA agent catching errors a combined agent would self-approve — produces meaningfully better output, orchestration earns its place. For most teams, this threshold is around 3+ distinct domains of expertise required in the same pipeline.
Q2What actually happens when an agent fails mid-pipeline? Does the whole crew crash?›
By default in CrewAI, an unhandled exception in an agent does propagate up and can crash the crew. This is why the error handling patterns in this module matter so much. Best practice: (1) Tools should return error messages, not raise exceptions. (2) Wrap crew.kickoff() in try/except. (3) Your QA agent should catch output quality issues and flag them rather than passing garbage forward. (4) For truly critical pipelines, use a dead letter queue pattern: if kickoff() raises an exception, log the inputs and retry asynchronously rather than crashing the user's request.
Q3What's the difference between orchestrating with LangChain directly vs using CrewAI on top of LangChain?›
LangChain gives you the primitives: LLMs, tools, memory, chains. Wiring multiple agents together with pure LangChain means writing the coordination logic yourself — how output passes between agents, when each agent runs, what format output should be in. CrewAI adds an opinionated layer on top: Agent (role + goal + backstory), Task (description + expected_output + context), Process (sequential/hierarchical). You trade flexibility for convention. For 80% of multi-agent use cases, CrewAI's conventions are the right choice — they encode patterns that work. For highly custom coordination logic that doesn't fit CrewAI's patterns, pure LangChain gives you more control. Most teams start with CrewAI and only drop to raw LangChain when they hit a specific limitation.
Q4Can I mix orchestration patterns in the same system? Sequential for some parts, hierarchical for others?›
Yes — and this is actually a common production pattern. A high-level orchestrator might use event-driven pattern to receive triggers (a webhook fires when a new ticket arrives), which then kicks off a sequential CrewAI crew for the actual processing, which has a hierarchical sub-process for quality review. The patterns apply at different levels of abstraction. In CrewAI terms: you can only use one Process per Crew, but you can design your overall system to have multiple crews with different processes, chained together via Python code at the application level.
Q5How do I prevent circular dependencies — Agent A waiting for B, B waiting for A?›
Circular dependencies in CrewAI are a design-time problem, not a runtime one — the framework doesn't automatically detect them. Prevention: always draw your task dependency graph before writing code. If you find yourself wanting to put context=[task_b] on Task A and context=[task_a] on Task B, that's a circular dependency — you need to restructure. Usually the fix is: merge the two tasks into one task on one agent, or introduce an intermediate task that one agent does first, breaking the cycle. If Agent A needs information from Agent B's work but Agent B also needs information from Agent A — that's a design problem, not a tool problem.
Q6Should I use callbacks or verbose logging for observability? What's the practical difference?›
Both — but for different purposes. verbose=2 on the Crew prints every thought, tool call, and observation to stdout. Use this during development to understand what your agents are doing. For production, use LangChain callbacks — specifically Langfuse (free tier available) or LangSmith — which capture structured traces you can query later. The key benefit: when something goes wrong in production 3 days later, you have a complete record of every agent thought, every tool call input/output, and every task result. You can't debug production agentic systems without structured traces. Set this up before you go live, not after the first production failure.

Scenario-Based Questions

Choose the right orchestration pattern. Justify it. Then see the reasoning.
Scenario 1The Due Diligence Pipeline
🔵 Situation

A VC firm wants AI to run initial due diligence on startup pitch decks. For each startup: analyse the deck, research the founders, analyse the market, and assess competition — then produce a 1-page investment memo.

🔴 Constraints

Founder research and market analysis are independent of each other. Both need to complete before the investment memo can be written. Speed matters — analysts review 20 pitches/week. Output must be consistent enough to compare across companies.

🟡 Your Call

Which orchestration pattern? What order do tasks run in? Which tasks could run in parallel to save time?

Pattern: Parallel fan-out → Sequential fan-in

Deck analysis runs first (sequential — everything else needs it). Then founder research and market analysis run in parallel (they're independent of each other). Competition analysis also runs in parallel with those two. Finally, the memo writer runs last, with context from all four previous tasks.

In CrewAI terms: implement as two sequential crews — Crew 1 does deck analysis (1 task). Crew 2 is kicked off with deck analysis output and has 3 tasks that could conceptually run in parallel (use async kickoff or run them as separate sequential tasks — the latency difference is usually small enough that simplicity wins). A final Memo Writer task takes context from all research tasks.

Consistency tip: put the exact memo format in expected_output of the final task: "Markdown sections: Team | Market | Product | Competition | Risk | Verdict (Pass/Watch/Invest)." This produces comparable memos across all 20 weekly pitches.
Scenario 2The Unreliable Data Source
🔵 Situation

You're building a price monitoring crew that checks competitor pricing daily. Your primary data source (a scraping tool) fails ~20% of the time due to anti-bot measures. When it fails, you have a secondary source that's less complete but reliable.

🔴 Constraints

The crew must always produce some output, even if partial. You need to know which source was used in the final report. Failover should be automatic — no human intervention for the daily run.

🟡 Your Call

How do you design the tool to handle failures? Where does fallback logic live — in the tool, the agent, or the task? What does your QA agent check for?

Fallback logic in the tool, not the agent.

Build one tool that internally tries the primary source, catches failure, and falls back to secondary — returning the result with a flag: "SOURCE: secondary | Data: [results]" vs "SOURCE: primary | Data: [results]". Never put fallback logic in the agent's instructions — agents apply instructions inconsistently. Put it in deterministic Python code in the tool.

The agent receives the result with the source flag included. The Analyst agent's task description should say: "Note which data source was used in your report." The QA agent's task checks: "Is the source clearly attributed in the report?" If the secondary source returned partial data, the QA agent flags "PARTIAL DATA: secondary source used, missing [X fields]" rather than silently passing incomplete output as complete.

This design means: tool failures are invisible to the orchestration layer, the agent always has something to work with, and the output always tells you what it's based on.
Scenario 3The Human Approval Gate
🔵 Situation

A marketing team wants a crew that generates social media posts. The crew should research trending topics, generate 5 post options, and schedule the winner. But a human must approve before scheduling.

🔴 Constraints

The approval step cannot be skipped under any circumstances. Unapproved content must never reach the scheduler. The human might reject all 5 options, requiring regeneration. The system must work asynchronously — crew runs overnight, human approves in the morning.

🟡 Your Call

Where does the approval gate sit in the architecture? How do you prevent the scheduling agent from running without approval? What happens on rejection?

Split into two crews with a human gate between them.

Crew 1 (runs overnight): Research Agent → Content Generator → saves to output_file="pending_posts.md". This crew produces drafts and stops. No scheduling agent in this crew — it can't reach the scheduler.

Human gate: a simple approval UI (or even a Slack message with approve/reject buttons) shows the 5 options. On approval, the selected post ID is passed to Crew 2. On rejection, Crew 1 is re-triggered with additional instructions: "The previous 5 options were rejected for [reason]. Generate 5 new options in a different style."

Crew 2 (runs after approval, triggered by human): Scheduling Agent takes the approved post ID and schedules it. This agent only runs because a human explicitly triggered it — the architectural separation makes it impossible to accidentally skip approval.

Never try to build approval logic inside a single crew with a conditional agent. The separation is not a workaround — it's the correct pattern for any workflow requiring a hard human gate.
Scenario 4The Overly Complex Crew
🔵 Situation

A developer shows you their 8-agent crew: a Coordinator, 2 Researchers, 2 Analysts, 1 Fact-Checker, 1 Writer, and 1 Reviewer. It takes 25 minutes to run, costs $2.40 per run, and the output quality is inconsistent.

🔴 Constraints

The final use case is: research a company and write a 400-word profile. The team needs this to run 50 times per day. Current cost: $120/day. Budget: $30/day.

🟡 Your Call

What's wrong with this design? How do you fix it? What's your target architecture to hit the budget constraint?

Classic over-engineering. Collapse to 2 agents.

The task — research a company, write a 400-word profile — does not need 8 agents. The developer confused "more agents = more quality" with "right agents for the task." Two Researchers doing the same thing is duplication. Two Analysts reviewing each other's work is overhead. Fact-Checker + Reviewer is two QA layers for a single 400-word output.

Target architecture: Researcher Agent (with web search tool, strong backstory focused on company intelligence) → Writer/QA Agent (writes AND self-reviews the profile). Two tasks, two agents, sequential. Estimated cost: $0.08/run × 50 = $4/day. Well under budget.

The quality inconsistency in the original design was likely caused by the 8-agent coordination overhead introducing drift — each handoff was a chance for context to be lost or reinterpreted. Fewer, cleaner handoffs with precise expected_output almost always beats more agents with vague coordination.

Rule: start with the minimum viable crew (1–2 agents). Add agents only when you can demonstrate that specialisation produces measurably better output for the specific task.

Anti-Patterns

Three orchestration mistakes that waste time, money, and sanity.
Anti-Pattern #1 — Over-Orchestration
🔴 Symptom

8-agent crew for a 400-word output. 25-minute run time. $2+ per execution. Inconsistent output quality despite the complexity.

🟡 Root Cause

Building from perceived sophistication rather than actual requirements. Adding agents because "more agents = more thorough," not because each agent genuinely outperforms a generalist at its specific task.

🟢 Fix

Start with 1–2 agents. Add a new agent only when you can demonstrate it produces measurably better output than the existing agents doing the same task. Build up, not down.

Anti-Pattern #2 — No Failure Handling
🔴 Symptom

Crew runs fine in dev, crashes randomly in production. One transient API failure takes down a 5-agent pipeline. Errors are silent — bad output reaches users without any flag.

🟡 Root Cause

Tools that raise exceptions instead of returning error messages. No retry logic around kickoff(). No QA agent as the final task to catch silent quality failures before output is used.

🟢 Fix

All tools return error strings on failure. Wrap kickoff() in try/except with 2 retries. Add a QA agent as the last task. Set max_iter on every agent. These four practices handle 95% of production failures gracefully.

Anti-Pattern #3 — Undefined Handoff Formats
🔴 Symptom

Downstream agents produce poor output. Agent 3 seems to "ignore" what Agent 2 found. Final report is missing information that was clearly in an earlier task's output.

🟡 Root Cause

Task expected_output doesn't specify a structured format. Agent 2 produces prose. Agent 3 receives a wall of text and can't reliably extract specific values it needs. Information gets lost in the summarisation.

🟢 Fix

Design output formats to match downstream input needs. If Agent 3 needs specific fields from Agent 2, Agent 2's expected_output should produce those exact fields in a parseable format: "Markdown table with columns: Field | Value | Source." Structure your handoffs like API contracts.
