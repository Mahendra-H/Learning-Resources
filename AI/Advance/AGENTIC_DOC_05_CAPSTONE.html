What You're Building

A Business Intelligence crew that compounds knowledge across sessions and ships in Docker.
🎯 The Product — Business Intelligence Engine

You give it a company name. It returns a full Business Intelligence report — market position, competitors, key people, financial health signals, and a strategic assessment. But here's what makes it real: it remembers everything it's ever researched. The second time you ask about a company in the same sector, it builds on what it already knows. The more you use it, the smarter it gets.

Four agents. MCP-backed persistent memory. LangChain web search tools. Sequential orchestration. Docker deployment. A FastAPI wrapper so any application can call it. This is not a demo — it's a system you can actually use or show in interviews.

Phase 1 — MCP Memory Layer

Build the persistent brain before you build the agents.
python · src/memory.py
import json, datetime, uuid
from pathlib     import Path
from langchain.tools       import tool
from sentence_transformers import SentenceTransformer
import lancedb

# ── Paths ──────────────────────────────────────────────────────
KV_PATH  = Path("./data/agent_memory.json")
DB_PATH  = "./data/semantic_db"
embedder = SentenceTransformer("all-MiniLM-L6-v2")
db       = lancedb.connect(DB_PATH)

def _kv(): return json.loads(KV_PATH.read_text()) if KV_PATH.exists() else {}
def _save(m): KV_PATH.write_text(json.dumps(m, indent=2))

@tool
def remember(key: str, value: str, ttl: str = "STANDARD") -> str:
  """Save a fact to persistent memory with a unique descriptive key.
  Key format: 'company_name_data_type' e.g. 'acme_corp_ceo', 'acme_corp_funding'.
  TTL: EPHEMERAL (7 days), STANDARD (90 days), PERMANENT.
  ALWAYS call this after finding important information about a company."""
  m = _kv()
  m[key] = {"value": value, "saved_at": datetime.datetime.now().isoformat(), "ttl": ttl}
  _save(m)
  return f"✅ Saved to memory: {key}"

@tool
def recall(key: str) -> str:
  """Retrieve a fact from persistent memory by its key.
  ALWAYS check this FIRST before researching — another agent may have found it already.
  Returns the value with its age, or NOT FOUND."""
  m = _kv()
  if key in m:
    e = m[key]; age = (datetime.datetime.now() - datetime.datetime.fromisoformat(e["saved_at"])).days
    stale = " ⚠️ VERIFY: data is over 30 days old" if age > 30 else ""
    return f"{e['value']} (saved {age} days ago{stale})"
  return f"NOT FOUND: '{key}' — research this fresh."

@tool
def list_memories(prefix: str = "") -> str:
  """List all memory keys, optionally filtered by prefix.
  ALWAYS call this at the start of a research task to see what is already known."""
  m = _kv()
  keys = [k for k in m if k.startswith(prefix)]
  return f"Memory contains {len(keys)} keys: {', '.join(keys)}" if keys else "No memories stored yet."

@tool
def remember_insight(content: str, tags: str = "") -> str:
  """Store a research insight or analysis finding in semantic memory.
  Use for paragraphs of analysis that should be retrievable by topic similarity.
  tags: comma-separated keywords e.g. 'acme,fintech,competitor'"""
  record = {"id": str(uuid.uuid4()), "text": content, "tags": tags,
            "saved_at": datetime.datetime.now().isoformat(),
            "vector": embedder.encode(content).tolist()}
  if "insights" in db.table_names(): db.open_table("insights").add([record])
  else: db.create_table("insights", data=[record])
  return f"✅ Insight stored in semantic memory."

@tool
def search_insights(query: str) -> str:
  """Search semantic memory for insights similar to the query.
  Use this to find relevant prior research before starting new analysis.
  Returns the top 3 most relevant prior insights."""
  if "insights" not in db.table_names(): return "No insights in memory yet."
  hits = db.open_table("insights").search(embedder.encode(query).tolist()).limit(3).to_pandas()
  return ("\n---\n".join(hits["text"].tolist()) if len(hits) > 0
          else "No relevant prior insights found.")

# Export all memory tools as a list for agent assignment
ALL_MEMORY_TOOLS = [remember, recall, list_memories, remember_insight, search_insights]

Phase 2 — Define the Agents

Four specialists. Each with a role, a goal, a detailed backstory, and exactly the tools they need.
python · src/agents.py
from crewai       import Agent
from crewai_tools import SerperDevTool, FileWriteTool
from langchain_openai import ChatOpenAI
from src.memory  import ALL_MEMORY_TOOLS, remember, recall, list_memories, search_insights

llm    = ChatOpenAI(model="gpt-4o-mini", temperature=0.2)
search = SerperDevTool()
writer_tool = FileWriteTool()

# Shared memory discipline instruction embedded in all backstories
MEMORY_RULE = """
WORK PROTOCOL: 1) Start every task by calling list_memories() then search_insights().
2) Use recall() for any specific fact before searching the web.
3) After finding important facts, call remember() immediately.
4) After completing analysis, call remember_insight() to save it."""

researcher = Agent(
  role="Senior Business Intelligence Researcher",
  goal="Find comprehensive, accurate, and current information about {company}",
  backstory=f"""You spent 12 years at McKinsey's research division before moving to an
independent intelligence firm. You're known for finding non-obvious insights —
the second-order facts others miss. You're disciplined: you never repeat research
that's already in memory, and you always save new findings so the team builds
on a compounding knowledge base. {MEMORY_RULE}""",
  tools=[search, *ALL_MEMORY_TOOLS],
  llm=llm, verbose=True, max_iter=15
)

analyst = Agent(
  role="Business Intelligence Analyst",
  goal="Extract meaningful patterns and strategic insights from research on {company}",
  backstory=f"""Former Goldman Sachs analyst with a speciality in competitive intelligence.
You turn raw facts into strategic narratives. You identify the 'so what' behind
the data — not just what a company does, but why it matters, what risks it faces,
and where its growth vectors are. You search memory first, always. {MEMORY_RULE}""",
  tools=[recall, list_memories, search_insights, remember_insight],
  llm=llm, verbose=True, max_iter=10
)

report_writer = Agent(
  role="Business Intelligence Report Writer",
  goal="Write a clear, structured, executive-ready BI report on {company}",
  backstory="""You've written 500+ executive intelligence reports for C-suite audiences.
Your reports are known for: clear structure, specific data (not vague claims),
actionable conclusions, and a tone that respects the reader's time. You never
use corporate filler. Every sentence earns its place.""",
  tools=[writer_tool],
  llm=llm, verbose=True, max_iter=8
)

reviewer = Agent(
  role="Quality Assurance Analyst",
  goal="Verify the BI report is accurate, complete, and ready for executive consumption",
  backstory="""You're the last set of eyes before a report reaches a C-suite client.
You check: Are all claims supported by data in the research? Are there gaps?
Is the tone appropriate? Is anything potentially misleading? If the report is
good, you approve it. If not, you return 'REVISION NEEDED: [specific issues]'.""",
  tools=[recall, search_insights],
  llm=llm, verbose=True, max_iter=6
)

Phase 3 — Define the Tasks

Precise expected_output on every task. This is where report quality is decided.
python · src/tasks.py
from crewai       import Task
from src.agents   import researcher, analyst, report_writer, reviewer

def build_tasks(company: str):
  """Build the 4-task pipeline for a given company."""

  research_task = Task(
    description=f"""Research {company} comprehensively.
    Before researching, ALWAYS:
    1. Call list_memories() — check if {company} has been researched before
    2. Call search_insights('{company}') — find any stored analysis
    3. Use recall('{company.lower().replace(' ', '_')}_overview') if it exists

    Then research and find: company overview, founding story, business model,
    revenue/funding, key executives (name + role + background), main products/services,
    customer segments, key competitors, recent news (last 90 days), and known challenges.

    Save EVERY finding with remember() using keys like:
    '{company.lower().replace(' ', '_')}_overview', '{company.lower().replace(' ', '_')}_ceo', etc.""",
    expected_output=f"""A structured research document containing:
    - Company overview (founded, HQ, employees, business model)
    - Financial snapshot (funding or revenue if public)
    - Key executives: Name | Role | Background (3–5 people)
    - Products/services: Name | Target customer | Differentiator
    - Top 3 competitors with one-line differentiation each
    - Recent news: 3–5 bullet points with dates
    - 2–3 known challenges or risks""",
    agent=researcher
  )

  analysis_task = Task(
    description=f"""Analyse the research on {company} and extract strategic intelligence.
    First call search_insights('{company}') to find any prior analysis in memory.
    Build on prior analysis — don't repeat what's already been assessed.

    Produce: SWOT analysis, competitive position, growth vectors (where is the company heading),
    key risks (regulatory, competitive, operational), and a 1-paragraph strategic assessment.""",
    expected_output="""Strategic analysis containing:
    - SWOT: 3 points per quadrant (Strengths, Weaknesses, Opportunities, Threats)
    - Competitive Position: where this company sits vs competitors (leader/challenger/niche)
    - Top 3 Growth Vectors: specific areas of likely expansion with rationale
    - Top 3 Risks: specific risks with likelihood and potential impact
    - Strategic Assessment: 1 paragraph synthesis — the 'so what'""",
    agent=analyst,
    context=[research_task]
  )

  writing_task = Task(
    description=f"""Write the executive Business Intelligence Report on {company}.
    Use all research and analysis from the context. Be specific — cite numbers,
    names, and dates. No vague statements. Write for a C-suite reader who has
    5 minutes, not 30 minutes. Save the report to outputs/{company.lower().replace(' ', '_')}_bi_report.md""",
    expected_output=f"""A complete markdown BI report with these exact sections:
    # Business Intelligence Report: {company}
    *Generated: [date] | Confidence: [HIGH/MEDIUM/LOW]*

    ## Executive Summary (3–4 sentences — the entire report in miniature)
    ## Company Overview (who they are, what they do, scale)
    ## Financial Snapshot (funding, revenue, valuation if available)
    ## Leadership (3–5 key people with roles and brief background)
    ## Products & Services (what they sell, to whom, why it matters)
    ## Competitive Landscape (position + top 3 competitors)
    ## SWOT Analysis (formatted table)
    ## Strategic Assessment (opportunities + risks + outlook)
    ## Sources & Confidence Notes""",
    agent=report_writer,
    context=[research_task, analysis_task],
    output_file=f"outputs/{company.lower().replace(' ', '_')}_bi_report.md"
  )

  review_task = Task(
    description=f"""Review the BI report on {company} before it is delivered.
    Check: Are all facts in the report supported by the research context?
    Are there gaps — sections that are vague or unsupported?
    Is the tone executive-appropriate? Any claims that could be misleading?
    If the report passes, return it with 'APPROVED' prepended.
    If not, return 'REVISION NEEDED: [list specific issues with line references]'.""",
    expected_output="""Either:
    'APPROVED' followed by the final report (copy it verbatim)
    OR
    'REVISION NEEDED: [numbered list of specific issues that must be fixed]'""",
    agent=reviewer,
    context=[research_task, analysis_task, writing_task]
  )

  return [research_task, analysis_task, writing_task, review_task]

Phase 4 — Assemble and Run

Wire it up. Run it. Verify memory persists across sessions.
python · src/crew.py
from crewai     import Crew, Process
from src.agents import researcher, analyst, report_writer, reviewer
from src.tasks  import build_tasks
from dotenv     import load_dotenv
import logging, sys, os

load_dotenv()
logging.basicConfig(level=logging.INFO, format="%(asctime)s %(levelname)s %(message)s")
logger = logging.getLogger(__name__)

def run_bi_crew(company: str, max_retries: int = 2) -> dict:
  """Run the BI crew for a company. Returns result dict."""
  logger.info(f"Starting BI crew for: {company}")
  tasks = build_tasks(company)

  crew = Crew(
    agents=[researcher, analyst, report_writer, reviewer],
    tasks=tasks,
    process=Process.sequential,
    verbose=2,
    memory=True        # CrewAI's built-in short-term memory across tasks
  )

  for attempt in range(max_retries + 1):
    try:
      result = crew.kickoff(inputs={"company": company})
      output_file = f"outputs/{company.lower().replace(' ', '_')}_bi_report.md"
      report_text = open(output_file).read() if os.path.exists(output_file) else str(result)
      logger.info(f"Crew completed successfully for {company}")
      return {"status": "success", "company": company, "report": report_text}
    except Exception as e:
      logger.warning(f"Attempt {attempt+1} failed: {e}")
      if attempt == max_retries:
        return {"status": "error", "company": company, "error": str(e)}

if __name__ == "__main__":
  company = sys.argv[1] if len(sys.argv) > 1 else "Stripe"
  result  = run_bi_crew(company)
  print(f"\nStatus: {result['status']}")
  if result["status"] == "success":
    print(f"\n{result['report'][:500]}...")

# Run: python -m src.crew "Stripe"
# Run: python -m src.crew "OpenAI"
# Second run on same company: agents check memory first — much faster!

Final Myths — Production Agentic Systems

Three things developers believe after building their first crew that experience disproves.
MYTH"Once the agents are good, the system is reliable. The hard work is building the agents."
REALITYIn production, the agents are table stakes. The hard work is everything around them: memory management (what to keep, when to expire, how to handle staleness), error handling (graceful degradation when a tool fails), observability (knowing which agent produced bad output at 3am), cost management (preventing runaway API spend from infinite loops), and quality assurance (catching hallucinated facts before they reach users). The crew from this module is perhaps 40% of the work needed to run it reliably at scale. Build the surrounding infrastructure before you go live.
MYTH"Persistent memory makes agents smarter over time. The longer you run it, the better it gets."
REALITYPersistent memory makes agents faster and more consistent over time — not necessarily smarter. Without a retention policy, it makes them slower and less accurate as stale, contradictory, and irrelevant entries accumulate and degrade search results. The memory store needs active management: TTL policies, periodic cleanup, staleness flagging, and confidence scoring. Treated as a passive accumulation layer, memory becomes a liability within 2–3 months of production use. Treated as a curated knowledge base with disciplined write standards and expiry policies, it compounds genuine value.
MYTH"You should automate as much as possible. Any step requiring human approval is a sign of insufficient AI capability."
REALITYHuman-in-the-loop checkpoints are a feature, not a failure. For any output that reaches customers, makes financial decisions, affects legal standing, or influences strategy — a human review gate is the appropriate architecture, not a workaround for AI limitations. The goal of agentic systems is not to remove humans from every decision but to remove humans from the decisions that don't require human judgment. The BI report in this module should be reviewed by a human before being sent to a board. That's correct system design. Building in review points also gives you data on where AI output needs improvement — which is far more valuable than shipping unreviewed output and finding out from an embarrassed client.

Series FAQ — The Big-Picture Questions

After five modules, these are the questions that pull everything together.
Q1How do I know when my agentic system is ready for production?›
Four gates. (1) Reliability gate: Run it 20 times on diverse inputs. Does it produce acceptable output at least 85% of the time without human intervention? (2) Failure gate: Deliberately break a tool. Does the system degrade gracefully or crash? (3) Cost gate: Run a simulated month of production load. Is the cost acceptable? (4) Quality gate: Have 3 domain experts review 10 outputs each. Do they judge them accurate and useful? If all four pass — ship it. If any fail — fix that first.
Q2My crew runs in dev but is too slow for production. What's the fastest way to reduce latency?›
In order of impact: (1) Upgrade to gpt-4o-mini if not already — it's 4–6× faster than gpt-4o with acceptable quality for most tasks. (2) Reduce max_iter — agents often iterate unnecessarily. Set max_iter=8 on most agents unless you have a specific reason for more. (3) Check memory hits — are agents doing web searches for things already in memory? If so, improve the memory check instructions in backstories. (4) Reduce tool call overhead — SerperDevTool returns large payloads. Use a custom tool that returns only the fields you need. (5) Parallelize independent tasks — if analysis and a secondary research task are independent, run them in parallel via async kickoff.
Q3How do CrewAI and MCP fit together with the rest of the AI landscape? Where does this series leave me?›
After this series you can build production multi-agent systems — which puts you ahead of the majority of developers who can only build single-LLM applications. The landscape context: LangChain provides the infrastructure (tools, memory, LLMs). CrewAI provides the coordination layer (multi-agent orchestration). MCP provides the context standard (how agents share knowledge). These three together cover the full stack for agentic AI. What this series doesn't cover: fine-tuning models (Hugging Face series), computer vision agents, voice agents, and the evaluation/testing discipline for agentic systems. These are the natural next topics after completing this series.
Q4Should I use CrewAI or build custom multi-agent systems from scratch? When does each make sense?›
CrewAI for 90% of production use cases. Build from scratch when: (1) your coordination logic is genuinely too complex for CrewAI's patterns (rare), (2) you need sub-second response times that CrewAI's overhead can't meet, or (3) you need extreme control over every token spent and have the engineering resources to maintain custom infrastructure. The argument for custom-built multi-agent systems is almost always premature optimization — teams spend months building what CrewAI provides out of the box, then discover their bottleneck was prompt quality, not framework flexibility.
Q5What's the actual career impact of knowing this stack? What jobs does this unlock?›
Concrete roles: AI Engineer, ML Engineer (applied), Solutions Architect at AI companies, Senior Developer at any company investing in AI automation. The specific skill of "can build production multi-agent systems" is genuinely rare in 2024–2025 — most developers can build chatbots; far fewer can build reliable autonomous workflows. Portfolio signals that matter: a deployed agentic system (GitHub + live URL > description), documented production metrics (cost/run, reliability %, latency), and code that shows proper error handling and memory management — not just the happy path.
Q6How do I evaluate agent output quality? "It seems good" isn't a production metric.›
Three evaluation approaches. (1) Criteria-based: Define explicit criteria for each task's output (completeness, accuracy, format compliance, tone). Score each run on each criterion (1–5). Track averages over time. (2) LLM-as-judge: Run a separate LLM call that compares agent output against a gold standard example and scores it. Consistent, scalable, correlates well with human judgment. (3) Human evaluation: For high-stakes outputs, have domain experts rate a sample monthly. Use their feedback to improve prompts and backstories. Combine all three: LLM-as-judge for continuous monitoring, human evaluation for calibration, criteria-based for pass/fail gating.
Q7My agents hallucinate facts. I can't use this in production if I can't trust the output. How do I fix it?›
Four interventions, in order of effectiveness. (1) Ground every claim in a source: Change expected_output to require "[fact] (Source: [URL/document])" for every specific claim. Agents can't cite sources for hallucinated facts — it forces them to only state what they can verify. (2) Add a Fact-Checker agent: A dedicated agent that takes the final report and attempts to verify each claim with a fresh web search. Flags anything it can't verify. (3) Reduce temperature: Set temperature=0.1 for research and analysis agents. Higher temperature produces more creative, less grounded output. (4) Use GPT-4o over GPT-4o-mini for the most critical outputs — the larger model halluculates less on factual tasks.
Q8I've finished the series. What should I build next to consolidate this knowledge?›
Three project tiers. Immediate (this week): Deploy the capstone crew on a free tier (Railway.app or Fly.io) so it has a public URL. Add Langfuse for observability. Run it on 10 real companies and document what breaks. Short-term (this month): Adapt the BI crew for a specific industry or use case you care about (legal research, investment analysis, competitive intelligence for your own company). Replace generic backstories with domain-specific ones. Measure quality improvement. Medium-term (this quarter): Build a crew that performs a complete business function — an automated sales intelligence system, a content generation pipeline, or a code review system — and get one real user outside yourself to use it weekly.

Capstone Scenario Questions

Five complete system design problems. Justify every decision.
Scenario 1Scale the BI Crew to 100 Companies/Day
🔵 Situation

A PE firm wants to run the BI crew on 100 companies every morning before market open. Results must be ready in 90 minutes. Currently one crew takes 8 minutes per company — that's 800 minutes total.

🔴 Constraints

Budget: <$50/day API cost. 90-minute window. Memory must be shared across all parallel runs. No duplicate research across companies in the same sector.

🟡 Your Call

How do you parallelise? How do you handle shared memory with concurrent writes? How do you prevent 10 crews all researching the fintech sector independently?

Batch parallelisation with sector pre-seeding.

Parallelisation: Run 10–15 crews simultaneously using asyncio or multiprocessing. With 15 parallel crews at 8 min each, 100 companies = ~54 minutes. Within budget.

Shared memory with concurrent writes: Replace JSON file with Redis for thread-safe concurrent reads/writes. Redis handles concurrency natively. All 15 parallel crews read from and write to the same Redis instance without conflicts.

Preventing duplicate sector research: Add a "Sector Pre-Seeder" step before the parallel runs: a single crew researches each sector (fintech, SaaS, healthcare) once and stores findings as sector_{name}_overview in memory. Individual company crews then call recall('sector_fintech_overview') and skip sector-level research entirely, focusing only on company-specific data. This alone reduces per-company research time by ~30%.
Scenario 2Confidential Client Reports
🔵 Situation

The same BI system is now used by 3 different clients. Client A's research on competitors cannot be visible to Client B — they're in the same industry and the research contains NDA-protected intelligence.

🔴 Constraints

Each client needs their own isolated memory namespace. Generic sector knowledge (market sizing, public data) should be shared across clients to reduce costs. The isolation must be technically enforced, not just policy-based.

🟡 Your Call

How do you restructure the memory system? What changes to the API? How do you verify isolation without violating client confidentiality?

Tenant-scoped memory tools with shared public namespace.

Memory restructure: Three namespaces per design: public/{key} (market data, public company info — all clients can read/write), client_a/{key} (Client A's private research), client_b/{key}. Each namespace in Redis uses a key prefix.

Tool restructure: Create client-scoped tool instances at API request time. remember_client_a() only writes to client_a/ or public/. recall_client_a() only reads from client_a/ and public/. The filtering is in Python, not in the LLM prompt.

API change: POST /analyze requires a client_id field in the request body. The crew is instantiated with client-scoped tools. Verification: create an integration test that runs Client A's crew then asserts that Client B's recall() tool cannot return Client A's private keys.
Scenario 3The Hallucinating Researcher
🔵 Situation

After 3 weeks in production, your team discovers that the Researcher agent occasionally fabricates revenue figures for private companies, citing plausible-sounding but non-existent sources. Three client reports have gone out with incorrect numbers.

🔴 Constraints

You can't stop production — 20 clients depend on daily reports. Fix must be deployable in <24 hours. The fix must reduce hallucination without significantly impacting report quality or latency.

🟡 Your Call

What's your immediate fix? What's your structural fix? How do you identify which past reports have bad data?

Immediate: citation requirement. Structural: dedicated fact-check agent.

Immediate fix (<2 hours): Change the Researcher's expected_output to require every numerical fact to include its source: "Revenue: $50M (Source: TechCrunch 2024-01-15, URL)." Add to backstory: "If you cannot find a verifiable source for a number, write 'Revenue: Not publicly available' — never estimate or approximate revenue figures." Deploy immediately.

Structural fix (<24 hours): Add a Fact-Check Agent as Task 5, between writing and review. Its job: take every numerical claim in the report and attempt to verify it with a fresh search. Return "VERIFIED: [claim]" or "UNVERIFIED: [claim] — source not found." The Reviewer agent then flags any UNVERIFIED items for human attention before the report is delivered.

Retrospective fix: Run the new fact-check agent against the last 3 weeks of stored reports (the output_file content). Identify which reports contain UNVERIFIED numerical claims. Contact affected clients proactively with corrections. This is the uncomfortable but correct response — it maintains trust and gives you data on which company types the agent hallucinates most (likely early-stage private companies with no public financial data).

Scenario 5The Memory That Outlived Its Data
🔵 Situation

After 6 months, a company you've tracked went bankrupt — but the memory still has "financial_health: STRONG, revenue: growing 40% YoY" from 8 months ago. A new BI report cites this, presenting the bankrupt company as financially healthy to a client who nearly makes an investment decision based on it.

🔴 Constraints

This nearly caused a $2M investment mistake. You need a systematic fix, not a one-off patch. The system processes 500 companies — you can't manually curate all memory.

🟡 Your Call

How do you prevent stale critical data from poisoning future reports? What's the architectural fix? What TTL policy should have been in place from day one?

Mandatory news freshness check + financial data TTL of 30 days maximum.

Root cause: Financial health data stored as STANDARD (90-day TTL) when it should have been EPHEMERAL (7-day TTL) or required freshness verification at every use.

Immediate fix: Identify all memory keys with category "financial" or "health" or "revenue" and expire them immediately. Add a weekly job that auto-expires any financial data older than 30 days regardless of TTL setting.

Architectural fix: Create a "High-Volatility" data category with 14-day TTL. Financial health, leadership changes, market position, and recent news all go in this category. "Low-Volatility" (founding date, HQ location, business model) stays at 90-day STANDARD. The Researcher's instructions: "For any financial or competitive position data, ALWAYS verify with a web search even if memory has a recent value — these can change rapidly."

Structural fix: Add a "News Freshness Check" as the first subtask of every research job: search for "[company name] news last 30 days" and check for bankruptcy, acquisition, leadership change, or financial crisis signals BEFORE retrieving any other memory. If such signals are found, invalidate all financial memory for that company and research fresh. This "canary check" catches the most dangerous staleness scenarios in under 30 seconds.
