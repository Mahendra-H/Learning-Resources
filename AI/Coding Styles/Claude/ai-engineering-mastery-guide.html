01The Engineering Mindset

Why discipline beats intuition at scale.

AI makes it possible to move fast without structure — for a while. The ceiling arrives when you try to maintain, secure, extend, or debug what you built without structure. This section explains exactly where the ceiling is and how to engineer past it.

Why Vibe Coding Fails at Enterprise Scale

Vibe coding works for prototypes. It fails when:

  • Context evaporates. AI has no memory across sessions. Every session starts cold. Without structured context files, you re-explain your architecture hundreds of times across hundreds of sessions — burning tokens and getting inconsistent output.
  • Output is unreviewed at scale. One unreviewed AI-generated function is a small risk. A hundred unreviewed functions across a production system is a security incident waiting to happen.
  • Scope explodes without specs. Without a spec, neither you nor the AI knows what "done" means. Every feature expands until you manually stop it — usually after it's already touched files it shouldn't have.
  • Technical debt compounds faster. Vibe-coded systems are built on assumptions that were never written down. When those assumptions change, the system breaks in ways you can't trace.

The 5 Pillars of Enterprise AI Engineering

📐
Architecture-First Mindset
You design before you build. Every feature begins with a plan — files to create, boundaries to respect, patterns to follow. AI implements the plan. It never defines it. The plan comes from you, from the spec, from the ADR.
📋
Specification as Contract
A spec is the contract between your intent and AI's execution. Without it, AI generates what it thinks you want. With it, AI generates what you need. The spec also defines "done" — without that, you can't review output.
🔍
Review-First Discipline
AI output is generated, not verified. It may be brilliant. It may contain a critical security flaw. You don't know until you review. Review prompts are not optional — they're the quality gate that makes AI-generated code trustworthy.
🔗
Incremental Delivery
One unit of work per prompt. One prompt per session step. One session step per chain link. This is what makes output reviewable. It's not slower — it's more accurate, and accuracy eliminates rework, which is the actual time sink.
👤
Human Governance
You own the requirements, the architecture decisions, the security constraints, and the production approval. AI accelerates execution within those bounds. The moment you let AI define what to build — not just how — you've delegated governance to a tool that has no accountability for the outcome.

The 5 Questions Before You Type Anything Run Every Session

Elite engineers run this checklist mentally before every Copilot session. It takes 60 seconds. It eliminates the five most expensive categories of AI-assisted development waste.

QUESTION 01
Do I have a spec?
specs/[feature]/requirements.md
YES → proceed to Q2 NO → stop. Write requirements.md + edge-cases.md + acceptance-tests.md first. Coding without a spec means AI invents requirements. Those inventions ship as bugs.
QUESTION 02
Do I have an architecture plan?
01-architect-plan.md
YES → proceed to Q3 NO → stop. Run architect plan prompt. Review output before writing any code. Architecture errors in code cost 10× more to fix than architecture errors in a plan.
QUESTION 03
Do tests exist for what I'm building?
03-test-writer.md
YES → proceed to Q4 NO → stop. Run the test-first prompt. Tests define "done." Without them there is no definition of correct. Tests written after implementation verify what the code does. Before — what it should do. Not the same.
QUESTION 04
Is my scope declared and minimal?
Scope: ONLY [file]
YES → proceed to Q5 NO → stop. Write it now: "Scope: ONLY [file]. Do NOT modify: [list]." Undeclared scope is an open invitation for AI to refactor things you didn't ask it to touch.
QUESTION 05
Is my .ai/ context current?
.ai/project-context.md
YES → open Copilot. Write the 3-line session opener. You are ready. NO → stop. Update the stale file first. 5-minute fix now prevents 6 weeks of inconsistent AI output. All five YES? You have everything needed for a clean, scoped, reviewable session.

03Saving Tokens

Every wasted token is a wasted decision.

Token waste is not just a cost problem — it's a quality problem. Long, repetitive, context-bloated sessions produce worse output. Lean, focused sessions with smart references produce better output. These seven strategies eliminate the most common sources of token waste.

1
Reference files — never paste their contents
↓ 80–95% savings
▾
❌ BAD — 400 tokens
This project uses FastAPI with Python 3.11. We use PostgreSQL with async SQLAlchemy. Auth is JWT via python-jose. Logging with structlog. PEP8 strictly enforced. No bare excepts. All models use Pydantic v2... [200 more lines of context]
✅ GOOD — 8 tokens
Read .ai/project-context.md before proceeding.
Rule: Copilot reads files you reference. You never need to paste their contents. Set it up once in copilot-instructions.md and it loads automatically for every session.
2
One task per session — never mix topics
↓ Improves quality 40%
▾
❌ BAD
Fix the refresh token bug, also while you're there add the rate limiter I mentioned earlier, and can you also review the JWT validator class?
✅ GOOD
# Session 1: Fix refresh token bug Task: Bug Spec: specs/bugfix-refresh-token/ Scope: ONLY src/auth/token_refresh.py # Session 2 (new session): Rate limiter # Session 3 (new session): JWT review
Why: Mixed sessions pollute the context. The model tries to hold multiple tasks in working memory simultaneously and does all of them worse. One session = one task = consistently better output.
3
Use the 3-line session opener — stop over-explaining
↓ 60% session startup waste
▾
❌ BAD — 150 tokens to start
OK so I'm working on the JWT middleware. This is a FastAPI project. We've been building out the auth layer. I need to add the validation logic. The spec is in specs/jwt-middleware but I also need to make sure it follows our coding style...
✅ GOOD — 12 tokens to start
Task: Feature Spec: specs/jwt-middleware/ Scope: src/middleware/jwt_validator.py
Why: copilot-instructions.md already loaded your project context. You only need to specify what's unique to this session — the task type, spec location, and scope. Three lines. Every time.
4
Use golden examples instead of style descriptions
↓ Eliminates style inconsistency
▾
❌ BAD — 300 tokens of style instruction
Use dependency injection pattern. All services should have an interface. Error handling must use custom exceptions. Logging must use structlog with context. All public methods need type hints. Docstrings in Google format... [200 more lines of style guide]
✅ GOOD — 6 tokens, perfect consistency
Reference: src/examples/ Follow implementation style exactly.
Why: A concrete example communicates style infinitely better than prose descriptions. src/examples/ is your golden reference. One line references it. Every implementation session uses it. Consistent output every time.
5
Close sessions before they drift — maximum 12–15 exchanges
↓ Prevents quality degradation
▾
❌ BAD — 40-message session
# Message 1-5: Planning # Message 6-15: Spec writing # Message 16-25: Implementation # Message 26-35: Bug fixing output # Message 36-40: Review # Result: model has deprioritised # message 1-10 context. Output # contradicts the original plan.
✅ GOOD — focused sessions
# Session 1 (10 messages): Plan only # Close → start fresh # Session 2 (8 messages): Spec only # Close → start fresh # Session 3 (12 messages): Implement # Close → start fresh # Each session: fresh context, sharp output
Why: LLMs don't treat all context equally. Long sessions cause the model to progressively deprioritise early messages. Your architecture plan from message 3 may be effectively invisible by message 35. Short focused sessions keep everything sharp.
6
Compress prompts — remove all filler, use structure
↓ 30–50% prompt length
▾
❌ BAD — verbose, unstructured
Hey, I was hoping you could help me implement a JWT validator. I want you to make sure it's really secure and follows our coding standards. It should probably validate the token signature and check the expiry. Thanks!
✅ GOOD — structured, precise
Scope: ONLY src/auth/jwt_validator.py Read: .ai/coding-standards.md Read: .ai/security-rules.md Spec: specs/jwt-middleware/requirements.md Tests: tests/auth/test_jwt_validator.py Goal: make all tests pass.
Why: Conversational filler tokens produce conversational output. Structured structured input produces structured, precise, implementation-ready output. Adopt the Key: Value format for all implementation prompts.
7
Scope declarations prevent re-work tokens
↓ Eliminates unwanted refactors
▾
❌ BAD — no scope, generates everywhere
Implement the token validator and make sure it integrates properly with the rest of the auth system.
✅ GOOD — declared boundary
Scope: ONLY src/auth/jwt_validator.py Do NOT modify: src/auth/__init__.py, src/middleware/auth.py, any test files
Why: Without scope declaration, AI "helpfully" refactors adjacent code. You then spend tokens reviewing, correcting, and reverting changes you never asked for. Scope declaration makes every output deterministic and minimal.
04Context Engineering

The quality of AI output equals the quality of context you provide.

Context is everything the AI knows about your project before you give it a task. Poor context produces generic output. Rich, structured, file-based context produces output that feels like it was written by a developer who has been on your project for months.

How LLMs Actually Consume Context

  • Position matters. Content at the beginning and end of the context window has higher influence than content in the middle. Put your most critical constraints at the top of every prompt. This is why copilot-instructions.md loads first.
  • Specificity beats volume. A precise 50-line project-context.md outperforms a bloated 500-line one. Include only what's architecturally relevant. Every irrelevant line dilutes the signal.
  • Structure improves retrieval. Headers, sections, and clearly labelled rules in your context files are easier for the model to "locate" during generation. Unstructured prose context produces unstructured output.
  • Consistency is cumulative. The more your context files use consistent naming, the more consistent the generated code becomes. If your context says "UserService" and your code says "user_service" — expect inconsistency.

Avoiding Context Overflow — 5 Strategies

Pasting large files into chatSession fills up fast. Early context gets deprioritised.Reference by path. Read .ai/project-context.md — never paste it.
Sessions spanning multiple phasesPhase 1 architecture bleeds into Phase 2 generationOne session per phase. New session = fresh context load via continuation opener.

06Output Quality

Better input. Better spec. Better output. Every time.

AI output quality is not random. It is directly proportional to the quality of what you provide: the context, the spec, the scope declaration, and the constraints. Improve these five input dimensions and output quality improves measurably.

The 5 Input Dimensions That Drive Output Quality

DimensionPoor Input → Poor OutputGood Input → Good Output
Context QualityStale or absent .ai/ files → generic, inconsistent patternsAccurate, current .ai/ files → implementation matches your actual codebase
Spec CompletenessVague requirements → AI invents behaviour you didn't wantSpecific requirements + edge cases → AI implements exactly what was specified
Scope PrecisionUndefined scope → AI touches files you didn't intendDeclared scope + exclusions → minimal, reviewable output
Test ExistenceNo tests → AI optimises for "runs" not "correct"Tests exist before implementation → AI optimises for "tests pass"
Constraint ClarityImplicit rules → AI uses its defaults, not yoursExplicit rules (security, logging, error handling) → AI follows your rules

The Review Loop — How to Iteratively Improve Output

1
Run code review prompt immediately
Paste the generated file into the code reviewer. Don't read it manually first — let the prompt produce structured findings. Read the findings, not the code, initially.
2
Triage findings by severity
Critical and High: fix immediately. Medium: review each one, fix or explicitly accept. Low: note for later. Never skip Critical findings. They exist for a reason.
3
Fix findings with scoped prompts
For each finding: "Fix finding #N: [description]. Scope: ONLY [file]. Do not change anything else." One finding per prompt. Combining fixes produces new issues.
4
Re-run security review after fixes
Fixes can introduce new issues. A quick re-run of the security review prompt after fixing Critical/High findings catches second-order problems before they reach production.
5
Update spec if gaps were found
If review found something the spec didn't cover — add it to the spec now. This prevents the same gap from appearing in the next feature that touches the same module.
The Compounding Effect
Every time you improve a spec, update a context file, or add a new review finding to your security checklist — you permanently improve the baseline quality of all future AI output. SDD practitioners get progressively better output over time, not just consistent output.
07Practical Scenarios

Three workflow archetypes — every task maps to one.

Every development task you face maps to one of three workflow types. Identify the type first — then follow the matching workflow. Part 2 of this playbook series has full step-by-step walkthroughs of each scenario with every prompt written out.

SCENARIO A — FEATURE
JWT Validation Middleware
A new, bounded feature touching 2–3 files. Under 2 days. All 6 chain steps apply.
  • Session 1: Architect plan (no code)
  • Session 2: Write all spec files
  • Session 3: Tests before implementation
  • Session 4–5: Scoped implementation units
  • Session 6: Code + security review
  • Session 7: ADR + docs + context update
SCENARIO B — BUG FIX
Intermittent Refresh Token 401
A defect requiring diagnosis before fixing. Chain: diagnose → spec → minimal fix → regress → review.
  • Write symptom spec (not cause)
  • Run RCA diagnosis prompt
  • Confirm root cause in actual code
  • Minimal fix — declared scope only
  • Regression tests for the exact scenario
  • Security review (auth-related = mandatory)
SCENARIO C — ARCHITECTURE
Centralized Auth Gateway
A multi-week, multi-phase change touching multiple services. ADR first. Phased execution.
  • Architecture analysis prompt
  • Write ADR before any code
  • Phase breakdown (5 phases)
  • Contract-first design (OpenAPI)
  • Run full chain per phase
  • Migration + rollback plan per phase

08AI Anti-Patterns

Eight ways experienced engineers still get this wrong.

These anti-patterns feel productive in the moment. They feel fast. Some even feel like the "smart" shortcut. They are not. Every one of them produces slower total delivery time when you count rework, debugging, security incidents, and the quiet accumulation of untested code. Click each one to see the bad prompt, the correct replacement, and why it matters.

💣The Omnibus Prompt — "Build everything in one shot"ANTI-PATTERN▾
❌ BAD
Build a complete authentication system with JWT validation, refresh tokens, session management, audit logging, rate limiting, and SSO support. Use our FastAPI stack with PostgreSQL.
✅ GOOD
Task: Feature Spec: specs/jwt-middleware/ Scope: ONLY src/auth/token_validator.py Read: .ai/coding-standards.md Tests: tests/auth/test_token_validator.py Goal: make tests pass.
Why it fails: Large prompts produce large, unreviewed output. Security holes hide in the parts you skim. The output will drift from your architecture. Tests won't exist. Total delivery time is always longer, not shorter.
📭The Specless Build — Implementing without requirementsANTI-PATTERN▾
❌ BAD
Add a rate limiter to the login endpoint. Make it secure and block users after too many failures.
✅ GOOD
Read: specs/rate-limiter/requirements.md Read: specs/rate-limiter/edge-cases.md Read: .ai/security-rules.md Scope: ONLY src/middleware/rate_limiter.py Tests: tests/middleware/test_rate_limiter.py
Why it fails: "Too many failures" is undefined. Per IP? Per account? What threshold? What lockout duration? What about distributed attacks? The spec answers all of these before a line is written. Without it, you inherit AI's assumptions as bugs.
👁️The Blind Ship — Committing AI output without reviewANTI-PATTERN▾
❌ BAD
# Looks good to me ✓ git add src/auth/ git commit -m "add jwt validation" git push origin main
✅ GOOD
# Before commit — always: # 1. Run: 06-code-reviewer.md prompt # 2. Run: 05-security-reviewer.md prompt # 3. Fix all Critical + High findings # 4. Run: pytest tests/auth/ # 5. Then: git add + commit
Why it fails: AI output may expose stack traces, log tokens, fail open on auth errors, or miss input validation. These are the most common security bugs in AI-generated code. Review prompts catch them in 5 minutes. Production incidents take days.
🌊The Scope Drift — "While you're at it, also..."ANTI-PATTERN▾
❌ BAD
Implement the token validator. Also while you're at it, refactor the auth middleware and update the session manager to use it too.
✅ GOOD
# Session 1: Scope: ONLY src/auth/token_validator.py Do NOT modify: any other file. # Session 2 (new session, after review): Scope: ONLY src/middleware/auth_middleware.py Do NOT modify: token_validator.py
Why it fails: "While you're at it" compounds review complexity multiplicatively. What starts as 30 lines becomes 150 lines across 4 files. You lose the ability to know what changed and why. Scope declaration prevents this entirely.
🕸️The Stale Brain — Outdated .ai/ context filesANTI-PATTERN▾
❌ BAD — project-context.md says "uses requests"
# Copilot generates (using old context): import requests response = requests.get(url, headers=headers) # ← you migrated to httpx 3 weeks ago # ← wrong library, wrong async pattern # ← doesn't follow new conventions
✅ GOOD — context updated same day
# After migration, same day: # .ai/project-context.md → update tech stack # .ai/coding-standards.md → add async pattern # .ai/architecture.md → update if structure changed # Rule: context update = part of "done"
Why it fails: A stale context file is worse than no context file. AI confidently generates wrong code because it trusts the context. Stale context degrades silently — there's no warning. Update context files the same day you make any relevant change.
🏗️The Implement-First Pattern — Skipping the planANTI-PATTERN▾
❌ BAD
I need centralised error handling across all API endpoints. Write the error handler and update all endpoint files to use it.
✅ GOOD
Read: .ai/architecture.md Plan centralised error handling: - Files to create vs modify - Error response schema design - Migration approach for existing handlers Do NOT generate code. Plan only.
Why it fails: Jumping to implementation without a plan means discovering architecture mistakes in code instead of in a plan. Fixing an architecture error in code costs 10× more than fixing it in a 5-minute planning session.
📦The Context Dump — Pasting entire files into chatANTI-PATTERN▾
❌ BAD — 400+ tokens per session
[pastes entire project-context.md] [pastes entire coding-standards.md] [pastes entire architecture.md] Now implement the token validator...
✅ GOOD — 15 tokens, same result
Read: .ai/project-context.md Read: .ai/coding-standards.md Read: .ai/architecture.md Implement: [task description]
Why it fails: Pasting file contents bloats the session with tokens that could have been a path reference. It also degrades output quality by pushing important later-context further from the model's attention window. Reference paths always — never paste.
🤖The AI Override — Delegating architecture decisions to AIANTI-PATTERN▾
❌ BAD
What's the best way to architect our authentication system? Just decide and implement it. I trust your judgement.

What Context Must Never Be Omitted

  • Security constraints. If your code touches authentication, input validation, or data access — always reference .ai/security-rules.md. AI will not apply security rules it doesn't know about. Missing this once is how a fail-open vulnerability ships.
  • Scope boundaries. Scope: ONLY [file] and Do NOT modify: [list] — every single implementation prompt. Without these, AI makes "helpful" changes to adjacent files that you didn't ask for and won't review.
  • Failure mode requirements. Explicitly state how the code must behave on failure. "Fail closed" vs "fail open" is not something AI will assume correctly — it needs to be stated. AI Workflow Orchestration

Orchestration is how elite engineers work on large codebases, long-running projects, and complex migrations without losing coherence. It is the discipline of managing AI sessions as a coordinated system — not as isolated conversations.

Managing Large Codebases — The Module-by-Module Strategy

Working on a large codebase? Never give AI a global view of everything and ask it to reason about the whole system. This is context overload that produces low-quality, over-general output. Instead: work module by module, with tight context scoping per module.

M1
Map the module boundaries first
Before starting any cross-module work: produce a module map. What does each module own? What are the public interfaces? Where are the seams? This map becomes the constraint set for all subsequent prompts. "Do NOT cross the boundary between module A and B" is only enforceable when you know where the boundary is.
M2
One module = one context set
Each module gets its own spec folder reference, its own context slice, and its own session sequence. When working on the auth module, your prompts reference specs/auth/ and .ai/architecture.md auth section only. Not the billing module. Not the reporting module. Module isolation prevents architectural contamination between sessions.
M3
Synchronise architecture.md between modules
After completing each module, update .ai/architecture.md to reflect any interface changes. The next module's sessions will reference this file — if it's stale, the next module will make incorrect assumptions about the interface it consumes. Architecture sync is not optional between modules.

The AI-Assisted Debugging Loop

Elite engineers use AI for structured debugging — not "fix this for me." They use AI to systematically narrow the hypothesis space. The human decides which hypothesis to test. The AI implements the test. This is faster and more accurate than asking AI to guess at a fix.

LOOP STEP 1
Observe
Human-led
Capture exact symptom, logs, conditions. Write it in specs/bugfix-[name]/requirements.md before opening Copilot.
LOOP STEP 2
Hypothesise
AI-assisted (RCA prompt)
Run RCA prompt → get ranked hypotheses. Human reviews and selects most plausible. Human inspects the actual code at the suggested location.
LOOP STEP 3
Isolate
Human decision
Human confirms hypothesis in code. Human writes micro-spec for the fix: scope, constraints, what must not change.
LOOP STEP 4
Fix
AI-assisted (scoped implementer)

LOOP STEP 5
Verify
AI + Human
Run regression test prompt → tests for exact scenario. Run security review if auth/input involved. Human runs full test suite. Human confirms fix.

Verification & Validation Discipline

The engineers who get burned by AI are not the ones who used it — they are the ones who trusted it without verification. AI can generate perfect-looking code that has a critical vulnerability in line 47. Validation discipline is what separates engineers who ship AI-generated code safely from those who ship incidents.

The Validation Ladder — Never Skip Rungs

Every generated implementation must climb the full validation ladder before shipping. Each rung catches a different class of failure.

1
Syntax & Linting
Does it compile? Does it pass your linter? Basic hygiene — but automated tools catch things the human eye misses. Run your linter before any other validation.
2
Unit Tests
Do all pre-written tests pass? The tests you wrote before implementation are the contract. If any fail — the implementation is wrong, not the tests. Never change a test to make implementation pass.
3
Code Quality Review
Run the code review prompt. Check: standards compliance, architecture alignment, error handling completeness, naming clarity, dead code. Fix all Critical + High before proceeding.
4
Security Review
Mandatory for any code touching auth, input handling, data access, or logging. Run the security review prompt AND the critic prompt. A senior engineer reads every finding — not just the Critical ones. Medium findings become Critical after deployment.
5
Architecture Validation
Does the implementation respect all module boundaries? Did it introduce any unexpected dependencies? Does it match the ADR decisions? Run a quick architecture review against .ai/architecture.md for anything that touched system boundaries.

Failure Modes & Recovery

These are not hypothetical. These are the real patterns that produce real incidents in real production systems. Every one of them is recoverable — but recovery is expensive. Prevention costs minutes. These stories exist so you recognise the pattern before it costs you a weekend.

🚨The Fail-Open Auth — A 2 AM IncidentSECURITY▾
"I used AI to implement JWT validation. The prompt was good — structured, scoped. The generated code looked clean. I ran my existing tests — they passed. Deployed to production. Three weeks later: monitoring alert. Some requests to protected endpoints were returning 200 with no auth header. Turned out the JWKS fetch had a try/catch that returned None on failure — and the calling code treated None as 'validation skipped, allow through.' The test hadn't caught it because I hadn't written a test for the JWKS-unreachable case."
Root cause: Edge case not in spec. JWKS failure path not defined. No test for the scenario. Security review not run.
Recovery: Hotfix in 45 minutes — add explicit fail-closed: if not claims: raise AuthenticationError("JWKS unavailable"). Root cause fix: add JWKS failure to edge-cases.md, add test, run security review on every auth change going forward.
Prevention: edge-cases.md must include all external dependency failure modes. Security review prompt explicitly checks for fail-open patterns. Critic prompt asks: "what happens if JWKS is unreachable?"
💸The "While You're At It" Refactor — 3 Days of DebuggingSCOPE CREEP▾
"I asked AI to implement a new caching layer for the token validator. Partway through I added: 'while you're at it, clean up the surrounding error handling — it looked messy.' The AI refactored 4 files. The generated code looked cleaner. Tests still passed. Two weeks later a subtle bug appeared in the billing module — turned out the 'cleaned up' error handling had silently swallowed an exception that was previously propagating correctly. Took 3 days to trace it back to the 4-file refactor I hadn't scoped properly."
Root cause: Scope declaration absent. "While you're at it" added 3 unreviewed modules to the output. No Do NOT modify list. Silent regression buried under apparent cleanliness.
Recovery: git bisect to find the commit. Revert the scope-creep changes. Re-implement only the intended caching layer with a proper scope declaration.
Prevention: "Scope: ONLY [file]. Do NOT modify: [list]" in every implementation prompt. "While you're at it" is banned from AI prompts. Every refactor is its own spec + its own session.
🕸️The Stale Context Drift — 6 Weeks of Inconsistent PatternsCONTEXT DECAY▾
"We migrated from requests to httpx for all HTTP calls in week 3 of the project. It was a one-afternoon job — we updated the code, updated tests, all good. We forgot to update project-context.md. For the next 6 weeks, every AI-generated HTTP call used requests — the old library. We had a mixed codebase: some new code using httpx (correctly, by hand), some AI-generated code using requests (from stale context), and no one noticed because both worked. The inconsistency only surfaced in a performance audit 6 weeks later."
Root cause: project-context.md not updated after library migration. No "update context" step in definition of done.
Recovery: audit all AI-generated code from weeks 4–9, replace requests with httpx, run test suite. Half-day work that could have been 5 minutes at migration time.
Prevention: "Update .ai/project-context.md" is part of definition of done for any library change. Monthly context file review scheduled. Team rule: no PR merges if context files are stale.
🤖The AI Architecture Decision — 4 Months of Technical DebtGOVERNANCE FAILURE▾
"I asked AI to design the data model for a new reporting feature. The generated schema looked elegant and complete. I implemented it. Four months later, a new business requirement arrived that the schema fundamentally couldn't support — the AI had made a normalisation decision that optimised for read performance but made a category of updates impossible without a full migration. The architectural assumption was buried in a design choice I'd accepted without fully evaluating."
Root cause: Architecture decision delegated to AI. No ADR written. No alternatives evaluated. Architectural assumption not surfaced and reviewed by human.
Recovery: 3-week migration to new schema in a live system. Dual-write period. 4x the work of writing an ADR and evaluating alternatives initially.
Prevention: AI presents architecture options with trade-offs — human decides. Every significant data model decision gets an ADR. Prompt: "Present 3 alternative approaches, each with long-term implications."
