01 The Confession

Let me describe the old me — honestly.

I was fast. Frighteningly fast. I could open GitHub Copilot, fire off a prompt, and have a new feature running in twenty minutes. My colleagues — if I had any on the project — would have been impressed. I was definitely impressed with myself.

I called it intuitive development. What it actually was, I now understand, was vibe coding. I was navigating by feeling, not by structure. I was asking the AI to build things I hadn't fully thought through. And for a while, it worked.

Here's what my daily development looked like in those days:

The Old Morning Routine
Open VS Code → Open Copilot Chat → Type a rough description of what I want → Accept what comes out → Run it → Tweak → Ship. Repeat tomorrow. Never write a spec. Never write tests first. Never ask "what does done actually look like?"

The first sign that something was wrong wasn't a catastrophic failure. It was a quiet, growing frustration. Every new Copilot session felt like meeting a brilliant colleague who had amnesia. I'd have to re-explain the project. Every. Single. Time.

"So, this is a FastAPI project. We use PostgreSQL. Authentication is JWT-based. The user model has these fields..."

I was burning tokens just to re-establish context. Then burning more tokens because my vague prompts produced output that needed fixing. Then burning even more tokens in back-and-forth clarification. I wasn't saving time — I was laundering it into a form I didn't recognise as waste.

Old Internal Voice
"I know this codebase inside out. I've been working on it for months. I don't need to write anything down. I'll just ask Copilot for the specific bit I need right now and it'll be fine."
New Internal Voice
"If I can't write it in a two-paragraph spec, I don't know it well enough to build it. The spec is not for Copilot — it's for me. It's how I discover the gaps in my own thinking before they become bugs."

The second sign came during a maintenance session. I had to fix a bug in a feature I'd written six weeks earlier. I looked at the code and thought — "who wrote this?" The answer was uncomfortable. I had. With Copilot. Without a spec. Without a clear plan. Just vibes.

It worked when I wrote it. Mostly. But it was untestable, undocumented, and written with a set of assumptions that I could no longer reconstruct six weeks later. I had shipped fast. But the cost was just deferred, not avoided.

02 The Wake-Up Moment

The night vibe coding stopped working.

Every vibe coder has a specific moment when the method collapses under its own weight. Mine came at 11 PM on a Tuesday. A feature that "worked fine" in development was silently corrupting data in a specific edge case I had never specced out. No test had caught it because there were no tests. The spec didn't exist to tell me the edge case mattered.

I opened Copilot to fix it. And then spent 40 minutes re-explaining the architecture, the data model, the business rule, and the security constraint — before I could even describe the bug. By the time I had provided enough context, I had a new problem: the conversation was so long and polluted with my re-explanation that the model's quality had degraded. I was getting answers that would have broken three other things.

The Real Cost of Vibe Coding
It's not the time you spend coding. It's the time you spend re-explaining, re-doing, debugging without context, and rewriting things you already wrote. Vibe coding is fast at the start and catastrophically slow at every subsequent step.

That night I started writing things down — not for Copilot, but for myself. A description of the system. The rules it must follow. The edge cases that matter. And I discovered something unsettling: I didn't know my own system as well as I thought I did.

Spec-Driven Development didn't start for me as a productivity technique. It started as a form of honesty. Writing a spec forces you to answer questions you've been avoiding. Questions like: what does this feature actually need to do? What happens when inputs are invalid? Who is allowed to do this? What does failure look like?

Vibe coding skips those questions. SDD demands them upfront. And that changes everything.

"Speed is not the enemy of quality. Vagueness is. And vibe coding is vagueness dressed up as velocity." — Something I had to learn the hard way
03 The Mindset Surgery

Five beliefs I had to surgically remove.

These aren't just habit changes. They're belief changes. Each one felt like giving something up before I understood what I was gaining.

Old Belief
"Speed is how I prove I'm productive. The faster I ship, the better I am at this."
→
New Belief
"Predictability is how I prove I'm productive. If I can ship reliably without regressions, I'm engineering. If I'm just moving fast, I'm gambling."
Old Belief
"AI is the developer. I describe what I want, it builds it."
→
New Belief
"I am the architect and engineer. AI is an accelerator for execution. It should never define what I build — only help me build it faster once I've defined it."
Old Belief
"Tests are for later. First make it work, then make it safe."
→
New Belief
"Tests define what 'works' means. Without them, 'it works' is just 'it ran once without crashing in front of me'."
Old Belief
"Docs and specs slow me down. They're overhead for big teams. I'm solo, I know what I'm building."
→
New Belief
"Docs and specs save me from future me. The person debugging this at 11 PM in six weeks is also me. I owe that person a spec."
Old Belief
"If it runs in prod, I did it right. Don't overthink it."
→
New Belief
"Running in prod means it hasn't failed yet — not that it won't. Architecture, security, and tests are what separate luck from engineering."
04 Old Habits vs New Disciplines

The five battles you'll fight every day.

These are the exact moments where the transition from vibe coding to SDD is won or lost. Expand each one to see the before, the after, and the actual prompts involved.

01 Opening Copilot before thinking ▾
The Old Way

The impulse hits. The IDE is open. Copilot Chat is right there. So I start typing. No plan, no scope, no definition of done. Just energy and intent.

Hey, I need to add a rate limiter to the API. Can you help me build that?
The New Way

Before Copilot opens, I answer three questions on paper: What exactly does this do? What does it not do? What does "done" mean? Then I write the spec. Then I open Copilot.

Read .ai/project-context.md. Spec is at specs/rate-limiter/. Read requirements.md and edge-cases.md. Generate implementation plan only. Do not write code yet.
Why this matters: Without a spec, Copilot is guessing at your requirements. It will build something — but it may not be what you needed. The spec is the contract that makes the output deterministic.
02 Giant prompts that try to do everything ▾
The Old Way

I describe the entire feature at once. The output is massive, unreviewed, touches dozens of files, and by the time I understand what was generated, half of it needs fixing.

Build me a complete user auth system with JWT, refresh tokens, forgot password, remember me, email verification, rate limiting, audit logs, and SSO via Google.
The New Way

One scoped task per prompt. Each task is small enough to review completely. Each output is validated before the next task begins. Ten small prompts beat one giant prompt every time.

Implement ONLY the JWT token validation function. Spec: specs/auth/requirements.md Scope: src/auth/token_validator.py Do not modify any other file.
Why this matters: Reviewing 20 lines is easy. Reviewing 300 lines of AI-generated code across 8 files is how security bugs enter production. Small scope = reviewable scope.
03 Accepting AI output without review ▾
The Old Way

The code looks right. It runs. I accept it. I move on. I discover a month later that error messages were exposing stack traces in production, and an input wasn't being validated.

# Looks good to me ✓ # Ship it git add . && git commit -m "add auth" git push origin main
The New Way

Every implementation gets run through the code review prompt AND the security review prompt before committing. AI output is treated as untrusted by default — because it is.

# After implementation: # 1. Run: .ai/prompts/06-code-reviewer.md # 2. Run: .ai/prompts/05-security-reviewer.md # 3. Fix all Critical + High findings # 4. Then commit
Why this matters: AI does not know your security rules unless you tell it. And even then, it may miss edge cases. You are the last line of defence. Act like it.
04 Re-explaining context every single session ▾
The Old Way

Every session starts the same way. A wall of context. Tech stack, conventions, what we've already built, what we can't change. Burning tokens before a single line of productive work.

This is a FastAPI app using Python 3.11. We use PostgreSQL, async SQLAlchemy, Pydantic v2. JWT auth via python-jose. Logging with structlog. PEP8 strictly enforced. No bare excepts. All models use dataclasses. Now, I need to...
The New Way

.github/copilot-instructions.md loads automatically. Context is always pre-loaded. Every session starts with the task — not with re-explanation. Token savings from day one.

# New session startup — all I say: Task: Feature Spec: specs/payment-webhook/ Scope: src/webhooks/processor.py # Copilot already knows everything else.
Why this matters: Context re-explanation is the single biggest source of wasted tokens. The .ai/ folder eliminates it permanently. Invest one hour setting it up — save hours every week.
05 Writing tests after (or never) ▾
The Old Way

Tests are a future problem. Right now, the feature works — I can see it working. I'll add tests "later." Later never comes. Three months later, I change something related and break this silently.

# TODO: add tests for this # (never happens) def process_payment(data): # 200 lines of untested logic
The New Way

Tests are written before implementation using the test-writer prompt. The spec's acceptance criteria become the test cases directly. Implementation exists to make the tests pass — not the other way around.

# Workflow order: # 1. Write spec (acceptance-tests.md) # 2. Run: .ai/prompts/03-test-writer.md # 3. Tests exist → NOW write implementation # 4. Implementation makes tests pass
Why this matters: Tests written before implementation define what the feature must do. Tests written after verify what the feature accidentally does. These are not the same thing.
05 The SDD Workflow

Every step earns the next one.

This is the workflow I now follow for every feature, bug fix, or enhancement. No step can be skipped. Each one funds the quality of what comes after it.

📋
STEP 01
Write the Requirements
Before anything else, write what the feature must do in plain language. Business requirement. Functional requirements (numbered). Non-functional requirements. Out of scope. Acceptance criteria. This is the contract — with yourself, with the AI, and with future you.
specs/feature-name/requirements.md Human owned
🗺️
STEP 02
Define the Edge Cases
What happens when the input is null? When the network is down? When the user doesn't have permission? When the database returns an unexpected row count? Edge cases written before implementation become test cases. Discovered after implementation become bugs.
specs/feature-name/edge-cases.md 02-spec-writer.md
📐
STEP 03
Get the Architecture Plan
Ask Copilot to plan, not to code. What files need to be created? What patterns will be used? What are the data models? What is the sequence of operations? The plan is reviewed and corrected before a single line of implementation is written. This is the cheapest moment to fix a mistake.
01-architect-plan.md Review the plan
🧪
STEP 04
Generate Tests First
Using the acceptance criteria and edge cases from the spec, generate the test suite before writing any implementation. The tests become the definition of done. They are the executable version of the requirements document.
03-test-writer.md tests/feature-name/
⚙️
STEP 05
Implement in Scoped Increments
One function, one service, one endpoint at a time. Each implementation prompt is tightly scoped with an explicit file boundary. No prompt modifies more than it declares. The golden examples in src/examples/ are the style reference for everything generated.
04-implementer.md src/examples/ ← style ref
🔍
STEP 06
Code Review + Security Review
Every implementation goes through two review passes before commit: a code quality review checking standards, architecture alignment, and test coverage — and a security review checking for OWASP API risks, missing validation, credential exposure, and fail-open patterns. Non-negotiable.
06-code-reviewer.md 05-security-reviewer.md
📝
STEP 07
Document + Record Decisions
Any significant architectural decision made during the feature gets an ADR — a short record of what was decided, why, and what was rejected. The documentation generator prompt creates module docs, sequence diagrams, and runbooks. Future you reads this in six months and actually understands the code.
07-adr-writer.md 08-doc-generator.md docs/adr/
06 The Transition Stages

This is a journey, not a switch.

Nobody becomes an SDD practitioner overnight. Here is exactly what each stage feels like, what it looks like from the outside, and how you know you've moved past it.

01
Week 1–2
The Skeptic
You've set up the workspace — the .ai/ folder, the spec templates, the prompt files. And now your first instinct is: this feels like more work, not less. You're right, in the short term. The first spec takes an hour. The old way would have had code in twenty minutes. This friction is real and it's uncomfortable.
"Why am I writing a requirements document for a three-line change? This is overkill. I'll just do this one quickly the old way and set up the process properly next time."
You skip specs for "small" tasks You use prompts inconsistently You still accept output without review
02
Week 2–4
The Experimenter
Something clicks. You use the context files and notice that Copilot's first-response quality has improved noticeably. You don't have to re-explain the tech stack. The architect plan prompt saves you from a bad design decision you wouldn't have caught until much later. You're starting to see what the structure buys you.
"OK, that worked better than I expected. The spec actually caught a gap in my thinking before I started coding. But I still feel slow. This takes deliberate effort every single time."
Specs are written for medium+ tasks Context files are actively maintained You run at least the code review prompt
03
Month 1–2
The Frustrated Practitioner
You're following the workflow but your specs are still not detailed enough. Copilot is generating implementations that miss edge cases you didn't write down. You're discovering that the bottleneck has shifted — it's no longer "did I describe the feature?" but "did I describe the constraints?" Your spec writing skills are the limiting factor now, not your coding skills. This is progress — even though it feels like a new type of failure.
"I wrote a spec. Copilot still got it wrong. Maybe this doesn't work. Or maybe my spec was too vague. The edge-cases file was almost empty. That's probably where the gaps are."
Specs exist but lack edge cases You're learning to write better acceptance criteria You start writing ADRs reactively after decisions
04
Month 2–3
The Believer
The breakthrough moment. You implement a feature from spec to tested implementation, and you look at the output and think — "this is exactly what I designed." No surprises. No off-target generation. No security review findings. And most significantly: it took less total time than comparable vibe-coded features, even including the spec time. The investment has started paying back.
"That was clean. The spec was tight, the architect plan was reviewed, the tests existed before the code, and the implementation basically fell out of the plan. I can see this now. I actually believe it works."
Full workflow on all features Regressions are rare and caught by tests New sessions start with task, not context re-explanation
05
Month 3+
The Architect
The workflow is internalized. You think in specs before you think in code. When a new feature request comes in — from a stakeholder, from a bug report, from your own roadmap — your first instinct is to write the requirements, not to open the IDE. You've started extending the workflow for your specific patterns. You're building your own prompt templates. Your .ai/ folder is a living document of real knowledge. The AI feels like a capable junior engineer following your design — exactly as it should be.
"I notice when I'm about to vibe code now. I catch myself before I open Copilot without a spec. The old way genuinely feels uncomfortable — it feels like building on sand. The structure isn't overhead anymore. It's how I think."
Specs written before IDE is opened Custom prompts for your patterns Context files are always accurate You write ADRs proactively
07 The Honest Relapses

You will slide back. Here's how to recover.

Every practitioner relapses. These are the five moments where the old habits will reassert themselves — and exactly what to do when they do.

🪤
The "Quick Fix" Trap
The moment
"It's just one small bug. I don't need a spec for this. Let me just fix the null check and move on." The fix expands. There are three related issues. An hour later you've touched six files without a plan and introduced a regression.
Recovery
If the fix touches more than one function, write a micro-spec. Three sentences. What's broken, what the fix is, what must not change. It takes two minutes and saves an hour.
⏰
The Deadline Override
The moment
Pressure is high. The feature is needed now. "There's no time for process — I'll do it properly next sprint." You vibe-code the feature under pressure and ship it. The weekend debugging session costs three times what the spec would have taken.
Recovery
The spec is not the overhead — the rework is the overhead. A ten-minute spec prevents a three-hour debug. When deadlines compress, compress the spec, not the absence of one.
🕸️
The Stale Context File
The moment
You added a new library three weeks ago. Changed the auth pattern. Renamed the module structure. But project-context.md still describes the old architecture. Copilot is now generating code that follows patterns you've already deprecated.
Recovery
After every significant change, spend five minutes reviewing .ai/project-context.md for accuracy. Add it to your definition of "done" for any architecture-touching task. A stale context file is worse than no context file.
📢
The Giant Prompt Relapse
The moment
"I know exactly what I want. Let me just describe the whole thing." The familiar comfort of the old way. The giant prompt. The massive output. The hours spent trying to understand what was generated and why half of it is wrong.
Recovery
Catch yourself when your prompt is more than five sentences. That's a sign you're trying to get Copilot to plan AND implement at the same time. Stop. Use the architect plan prompt first. Plan then implement.
🎭
The Solo Dev Overconfidence
The moment
"I know this codebase. I built the whole thing. I don't need to document my decisions." Then you onboard nothing — not even your future self — and six months later you can't reconstruct why a critical architectural choice was made the way it was.
Recovery
The audience for your ADRs and specs is not other engineers. It's you, in six months, with full amnesia about this decision. Write it for that person. They will thank you at 11 PM on a Tuesday.
🔄
The "New Project Energy" Skip
The moment
New project. Clean slate. The energy is high and the context is all in your head right now. "I'll set up the SDD workspace once I have something working." You never do. Three months later it's a second project without structure.
Recovery
Run the COSTAR setup prompt before writing the first line of code. The one hour you invest in setup returns fifty times over the project lifetime. A new project is the best moment for structure — not the worst.
08 The Payoff

What engineering actually feels like.

Three months in, the workflow is internalized. And the change is not just in the output — it's in the feeling of the work itself. I no longer dread opening a project I haven't touched in a month. I open the .ai/ folder, re-read the architecture doc and recent ADRs, and I'm back in context within minutes.

When a new feature arrives, my first instinct is to open a spec file and start writing. Not to open Copilot. The spec is the first tool. Copilot is the second. This ordering is everything.

I no longer have the midnight debugging sessions that used to define my worst weeks. Not because my code is perfect — but because tests catch regressions before they reach production, security reviews catch vulnerabilities before they become incidents, and scoped implementations mean that when something breaks, I know exactly what changed and why.

~60%
Reduction in tokens spent on context re-explanation per session
~3×
Fewer implementation retries when spec-driven vs vibe coded
Zero
Security issues shipped to prod in the months after adopting review prompts

The version of me that thought speed was the goal would look at my workflow now and say: "that's so much overhead." The version of me that has worked this way for months knows that what looks like overhead is actually the work. The real work. The planning, the spec, the review — these are not delays before development. They are development.

Vibe coding gives you the feeling of moving fast. Spec-Driven Development gives you the reality of actually arriving.

The North Star
Mastery of SDD doesn't mean you stop using AI or that you slow down. It means you've made the structured workflow so natural that the overhead disappears — and what remains is genuinely fast, high-quality, reviewable, secure engineering. The craft and the speed become the same thing.
Before SDD — The Vibe Coder
"I have a new feature. Let me open Copilot and describe what I want and see what comes out. It'll probably work. If it doesn't I'll fix it. How long could it take?"
After SDD — The Architect
"I have a new feature. Before I open anything, I need to know: what exactly does it do, what are the edge cases, what does done look like, and what must not change? Give me twenty minutes to write that down. Then we build."
⚡ Quick Reference

Your SDD cheat sheet.

New Feature
  • Copy specs/_template/ → specs/feature-name/
  • Prompt: 02-spec-writer.md to generate spec files
  • Prompt: 01-architect-plan.md — plan only, no code yet
  • Prompt: 03-test-writer.md — tests before code
  • Prompt: 04-implementer.md — one scoped task at a time
  • Prompt: 06-code-reviewer.md → fix findings
  • Prompt: 05-security-reviewer.md → fix Critical/High
  • Prompt: 08-doc-generator.md → document it
Bug Fix
  • Open bug-fix-skill.md — read pre-conditions
  • Describe symptom to Copilot — ask for diagnosis only
  • Review diagnosis, correct if wrong
  • If fix is in 1 function: use 04-implementer.md
  • If fix touches architecture: write micro-spec first
  • Always run 05-security-reviewer.md after auth/input fixes
Session Startup
  • .github/copilot-instructions.md auto-loads — no context pasting
  • State: Task type (Feature / Bug / Enhancement)
  • State: Spec path specs/feature-name/
  • State: Scope (exact files in play)
  • Nothing else needed. Proceed to work.
Architecture Decision
  • Prompt: 07-adr-writer.md with decision title
  • Review Context and Consequences sections carefully
  • Save to docs/adr/ADR-XXX-title.md
  • Update docs/adr/ADR-000-index.md
  • Update .ai/architecture.md if boundaries changed
Non-Negotiables (Never Skip)
  • No implementation without a spec
  • No commit without code review prompt
  • No commit without security review on auth/input/data changes
  • No hardcoded credentials or config values
  • No secrets or PII in logs
  • No production code without structured logging
Keep Your Context Alive
  • New library added → update project-context.md
  • Architecture changed → write ADR + update architecture.md
  • New naming convention → update coding-standards.md
  • New security rule → update security-rules.md
  • Monthly: review all .ai/ files for staleness
