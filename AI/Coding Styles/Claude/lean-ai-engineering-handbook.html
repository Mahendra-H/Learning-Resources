§ 01
Minimal High-Value Repository Structure
One folder addition. Maximum AI context reuse.

The only structural addition to your existing repo is a .ai/ folder. Everything else stays as-is. This folder IS the project memory for AI — not for humans during development.

§ 02
Core AI Context Files
High signal. Low noise. Token-efficient by design.

📄 .ai/context.md — Always Loaded

Single source of truth for architecture, constraints, and patterns. Target: 300–500 lines max. This is an AI instruction file, not documentation.

✓ Put Here
  • Stack and runtime versions
  • Module boundaries and dependency rules
  • Key architectural decisions + WHY
  • Coding conventions (naming, error handling, async)
  • Explicitly banned anti-patterns
  • Auth/security constraints
  • Performance constraints
✗ Never Put Here
  • Feature requirements
  • Bug descriptions
  • Sprint or ticket state
  • Endpoint-level implementation details
  • Completed work history
  • Meeting notes
markdown .ai/context.md — example structure ALWAYS LOAD 
## Stack
- Runtime: Node 20 + TypeScript 5 (strict mode)
- HTTP: Express 4, no class-based controllers
- DB: Postgres via pg-promise, NO ORM
- Auth: JWT, RS256, validated at middleware layer only

## Module Rules
- services/ — pure business logic, no HTTP types
- routes/  — thin, delegates to services immediately
- No circular imports. services never import from routes.

## Error Handling
- All async handlers wrapped in asyncHandler(fn)
- Never throw raw strings.
  Always: throw new AppError(code, message, statusCode)
- 4xx = client fault, 5xx = our fault. Never conflate.

## Forbidden Patterns
- No `any` casts without // ts-ignore justification comment
- No console.log in production paths — use logger.info/warn/error
- No synchronous file I/O in request handlers

📄 .ai/state.md — Always Loaded, Volatile

Current project reality. AI drift happens when this is stale. Update at end of every session. Target: 50–100 lines max.

markdown .ai/state.md — example structure ALWAYS LOAD 
## Active Work
Branch: feat/payment-webhooks
Focus: Stripe webhook ingestion → order state machine

## Known Issues

- Email service times out >30s, workaround: fire-and-forget with queue

## Recent Schema Changes

## Current Constraints

📄 .ai/api.md — On Demand

External and internal API contracts. Load only when touching integrations. Target: 200–400 lines max.

Load WhenDon't Load When
Touching Stripe/external APIsPure business logic changes
Designing new endpointsDatabase schema work only

📄 .github/copilot-instructions.md — Auto-Loaded

GitHub Copilot reads this file automatically. Mirror only the coding rules section from context.md. Keep under 150 lines — every wasted line costs Copilot accuracy.

◈ Key insight: context.md is an AI instruction file. state.md is a volatile notepad. prompts.md is a macro library. None of these are for humans to read during development — they're machine-readable context that happens to be in plain text.
§ 03
Context Loading Strategy
Targeted loading. Zero token flooding.
✓ Always preload (every session): context.md + state.md only. ~600 lines. Everything AI needs to behave project-aware.

Load on Demand

Task TypeAdditional Context to Load
API work.ai/api.md
Bug fixRelevant source file + error log only
RefactorOnly the module being refactored
New featureSpec only (inline, not a file)
DB workSchema file or relevant migration
Code reviewThe specific file being reviewed

Never Load

✗ Always Omit
  • Full codebase dumps or entire src/ folders
  • Completed feature specs from previous sprints
  • Test files (unless debugging tests specifically)
  • node_modules, lock files, build output
  • Previous conversation transcripts

Good vs. Bad Context Loading

✗ BAD — Token flood, low signal

Also here's my README, package.json,

Fix with DB-level lock or idempotency key.

service layer only (no HTTP types).

§ 04
Feature Development Workflow
Plan → Validate → Break down → Implement → Review. Never skip phases.
◈ Phase order is non-negotiable. Never skip to implementation. One planning prompt saves 10 correction prompts. Never implement before architecture validation.
New Feature Workflow
▼

Input: 1-paragraph description + load context.md + state.md

1
Architecture Prompt
Validate architecture before any code. Get module touch points, data flow, and risks.
2
Task Breakdown Prompt
Generate ordered implementation task list from the validated architecture. Max 8 tasks.
3
Implement Per Task
One task per prompt. Attach specific file. Follow existing patterns exactly.
4
Rule-Anchored Review
Review against rules from context.md. Request: issues only, no praise.

Step 1 — Architecture Prompt

prompt COPY & USE 
[context.md] [state.md]
Feature: [one paragraph]
Output:
1. Which existing modules are touched
2. New modules/functions needed
3. Data flow (3–5 steps max)
4. Risks/constraints given current architecture
No code yet.

Step 2 — Task Breakdown Prompt

prompt COPY & USE 
Given that architecture plan:
Break into ordered implementation tasks.
Format: [ ] Task — file — acceptance condition
Max 8 tasks. Flag any blocked by external dependency.

Step 3 — Implementation Prompt (per task)

prompt COPY & USE 
Task: [single task from list]
File: src/services/payment.service.ts
Constraints: [from context.md]
Implement only this task. Follow existing patterns exactly.

Step 4 — Review Prompt

prompt COPY & USE 
Review this implementation against these rules:
[paste context.md rules section]
Check: error handling, type safety, pattern compliance, edge cases.
Output: issues only. No praise.
Format: line — issue — fix
Bug Fix Workflow
▼
prompt COPY & USE 
[context.md] [state.md]
Bug: [symptom]
File: [file]
Error/log: [paste]

1. Root cause (not symptoms)
2. Minimal fix
3. Regression risk
4. Any related code with same bug
Refactor Workflow
▼
prompt COPY & USE 
[context.md]
Refactor target: [file or function]
Goal: [specific — "extract DB calls into repository layer"]
Constraint: no behavior changes, tests must still pass
Output: refactored code + list of what changed and why
Enhancement Workflow
▼

Same as New Feature workflow, but start with this preamble:

prompt COPY & USE 
Existing behavior: [describe clearly]
Enhancement: [what changes specifically]
What must NOT change: [list every invariant]

[then follow New Feature steps 1–4]

§ 06
Token Optimization Strategies
Less tokens. Better output. No re-explanation.

Core Techniques

1
Anchor to context files, never inline re-explain
Use [context.md loaded] instead of re-pasting your entire stack and conventions in every prompt.
2
Scope brutally
Bad: "Review my payment service"   Good: "Review error handling in processRefund() only"
3
Specify output format upfront
Add Output: numbered list, max 5 items, no explanations unless critical to every prompt.
4
Use negative constraints
Do NOT: suggest ORM, add logging beyond existing pattern, change function signatures — eliminates the most common AI drift.
5
One task per prompt
Combining design + implement + review produces mediocre output for all three. Sequence them.
6
Kill hallucination with file anchors
Always attach the actual file. Never describe code from memory. AI hallucinates descriptions, not attached files.
7
State.md as drift prevention
Update weekly minimum. A fresh read resets AI understanding to current reality without re-explaining history.

Prompt Compression Reference

Verbose (wasteful)Compressed (optimal)
"Can you please help me understand how I should go about implementing...""Implement: [task]"
"Make sure to follow our coding standards and patterns""[context.md loaded]"
"Don't use any ORMs or external libraries we haven't approved""Constraints: see context.md §Forbidden Patterns"
"Here's the full file, somewhere in here there's a bug...""Bug in processRefund(), lines 45–78 [attached]"

Multiple projects. Reusable context. Minimal switching.

Copilot Chat Context Strategy

◈ Open .ai/context.md + .ai/state.md as pinned tabs before every Copilot Chat session. Use #file:context.md + #file:state.md in Copilot Chat to attach them directly. Use @workspace sparingly — it scans everything and dilutes context. Prefer #file: for targeted attachment.

Project Memory Strategy

✓ state.md IS your project memory. Update it at the end of each working session — 5 minutes max. It replaces: mental context juggling, re-reading old conversations, re-explaining state to AI next session.
§ 08
Final Recommended Operating Model
Lean. Fast. Reusable. Low-maintenance. High-productivity.

Daily Workflow

Morning
≤ 5 min
Update state.md with today's focus
Open context.md + state.md as pinned tabs
Feature Work
Main session
1. Run architecture prompt → validate before writing code
2. Generate task list → work one task at a time
3. Implement with scoped file-anchored prompts
4. Review with rule-anchored review prompt
End of Day
≤ 5 min
Update state.md: what changed, what's broken, decisions made

The Operating Contract

Never explain architecture inline
It lives in context.md — eliminates repeated tokens
Never implement before planning
Catches architectural mistakes before they compound into rework
One task per prompt
Better output quality every time. Never combine design + implement + review.
Attach files, don't describe them
AI hallucinates on descriptions. File attachments ground responses in reality.
Update state.md weekly minimum
Prevents AI drift from stale context — the #1 cause of inconsistent output
Negative constraints in every prompt
Stops the most common unwanted AI suggestions before they appear
Review with rules, not vibes
Consistent, actionable, reproducible feedback against concrete criteria

Maintenance Overhead

FileUpdate FrequencyTime
context.mdWhen architecture changes15 min, rarely
state.mdEnd of each session5 min, daily
api.mdWhen integrations change10 min, occasionally
prompts.mdWhen better patterns emergeAs needed
✓ Total: ~5 min/day. Everything else is eliminated repetition, token waste, and inconsistent AI output. This system does not scale by adding files — it scales by ruthlessly keeping only what's necessary.

What This System Is Not

Not a documentation system. Not a project management tool. Not a sprint planning framework. context.md is an AI instruction file. state.md is a volatile notepad. prompts.md is a macro library. None of these are for humans to read during development — they're machine-readable context that happens to be in plain text.
