Concepts Explained

Plain-English definitions for every term you will encounter. Written for a strong enterprise engineer with zero ML background.

Ollama
A runtime for running LLMs locally — like a Docker daemon but for AI. ollama pull qwen3:8b downloads the model, ollama run qwen3:8b starts it. REST API at localhost:11434 for programmatic use.
Open WebUI
Web-based chat UI running in Docker that connects to your local Ollama. Looks like ChatGPT but everything stays on your machine. Supports document upload, multiple models, history, and system prompts.
GGUF
File format for quantized LLM weights. Like a .jar for Java — bundles everything the model needs. Models on Hugging Face are distributed as .gguf files. Ollama handles these automatically on pull.
Quantization
Compressing model weights from 32-bit floats to 4-bit integers. 7B model at full precision = 28 GB VRAM. At Q4_K_M = 4.5 GB with minimal quality loss. Essential for laptop inference. Q4_K_M is the sweet spot.
Embeddings
Dense numerical vectors encoding text meaning. "Access denied" and "permission rejected" land near each other in vector space. Used for semantic search — finding similar documents without keyword matching.
Vector Database
Database built for similarity search on embedding vectors. Given a query vector, finds N most similar stored vectors in milliseconds. ChromaDB (local), Pinecone (cloud), Qdrant. Core storage layer of any RAG system.
RAG
Retrieval-Augmented Generation. Find relevant docs using embeddings, include them in the LLM prompt, model answers based on your data not just its training. How you make AI know your private internal documentation.
AI Agents
LLMs with access to tools (functions). Model sees available tools, decides which to call and with what args, you execute the function, return the result, loop continues until task complete. LLM writing scripts it then runs.
MCP
Model Context Protocol. Anthropic JSON-RPC standard for connecting AI to external tools. Write an MCP server exposing your IAM APIs as tools. Any MCP client (Claude Desktop, Open WebUI plugin) calls them. Growing fast.
LangChain
Python/JS framework for LLM apps. Chains (sequences of LLM calls), agents (LLMs with tools), memory, data loaders. Heavy for simple tasks — prefer LlamaIndex for RAG, LangChain for complex agents.
LlamaIndex
Python framework focused on RAG and document indexing. Simpler than LangChain for "connect docs to LLM". Built-in Ollama support. Your Phase 3 primary tool. Start here, graduate to LangChain for agents.
Context Window
Max text an LLM processes in one API call, measured in tokens (~0.75 words each). Qwen3 8B = 32k tokens. Everything you send — system prompt, history, retrieved docs, question — must fit. Managing it is a core skill.

Installation Order

Follow these steps in exact order. Each one has a clear success checkpoint — do not move to the next step until the checkpoint passes. All commands are for Windows 11 PowerShell or Command Prompt unless noted.

1
Install Ollama for Windows
Ollama is the foundation. Everything else — Open WebUI, your code, LlamaIndex — talks to Ollama. Install this before anything else.
Step 1 — Download
Go to: https://ollama.com Click "Download for Windows" Run the OllamaSetup.exe installer Click through the wizard (no special options needed) Ollama installs as a background Windows service and adds itself to PATH
Step 2 — Verify installation
ollama --version # Expected output: ollama version 0.x.x
Step 3 — Pull your first model (this downloads ~5.2 GB)
ollama pull qwen3:8b # You will see a progress bar. Takes 5–15 min on home broadband. # "success" message appears when done.
Step 4 — Pull the embedding model
ollama pull nomic-embed-text # Much smaller (~274 MB). Downloads quickly. # This is needed for all RAG work in Phase 3.
Step 5 — Pull the fast model
ollama pull phi4-mini # ~2.5 GB. Used when you want faster responses.
Step 6 — Run a quick test
ollama run qwen3:8b "What is OAuth2 in one sentence?" # Model should respond within 30 seconds. # Type /bye to exit the interactive prompt.
Step 7 — Verify the REST API is running
# In a browser, visit: http://localhost:11434 # You should see: Ollama is running # Or via PowerShell: Invoke-WebRequest -Uri http://localhost:11434 | Select-Object -ExpandProperty Content
Ollama is running. ollama list shows qwen3:8b, phi4-mini, and nomic-embed-text. REST API responds at localhost:11434.
If ollama is not recognised as a command: close and reopen your terminal. Windows PATH update needs a new shell session. If still failing, reboot.
If model download stalls or fails: check your internet connection. Retry with ollama pull qwen3:8b — it resumes partial downloads automatically.
2
Install Docker Desktop
Docker is needed to run Open WebUI (Step 3) and later ChromaDB as a standalone server. You already understand Docker — this step should be quick.
Step 1 — Download and install
Go to: https://www.docker.com/products/docker-desktop Download "Docker Desktop for Windows" Run the installer. When prompted: ✓ Check "Use WSL 2 instead of Hyper-V" (recommended) ✓ Check "Add shortcut to desktop" REBOOT your machine after install — this is required.
Step 2 — Enable WSL2 (if prompted on first launch)
# Docker Desktop will show a dialog if WSL2 is not installed. # Click "Install WSL2 kernel update" and follow the link. # Download and run the WSL2 kernel update package from Microsoft. # Restart Docker Desktop after.
Step 3 — Verify Docker is working
docker --version # Expected: Docker version 26.x.x or higher docker run hello-world # Expected: "Hello from Docker!" message printed # If you see this, Docker is working correctly.
Step 4 — Verify Docker can reach the internet
docker pull alpine # Should pull a tiny Linux image. Confirms Docker networking is working.
Docker Desktop is running (whale icon in system tray). docker run hello-world prints the success message.
Docker Desktop requires Hyper-V or WSL2. On Windows 11 Home, WSL2 is the only option — make sure to install the WSL2 kernel update when prompted. Without this, Docker won't start.
If Docker says "WSL2 kernel update required": Download the kernel update from aka.ms/wsl2kernel, install it, then restart Docker Desktop.
3
Run Open WebUI in Docker
Open WebUI gives you a ChatGPT-like interface pointing at your local Ollama models. Once this is running you have a fully functional local AI chat environment. Takes about 2 minutes to set up.
Step 1 — Run the container (copy this entire command exactly)
docker run -d ` -p 3000:8080 ` --add-host=host.docker.internal:host-gateway ` -v open-webui:/app/backend/data ` --name open-webui ` --restart always ` ghcr.io/open-webui/open-webui:main # The backticks (`) are PowerShell line continuation. # If using CMD instead of PowerShell, replace ` with ^ # First run will download ~1.5 GB image. Wait for it to finish.
Step 2 — Verify the container is running
docker ps # You should see "open-webui" in the list with status "Up X seconds"
Step 3 — Open in browser and create admin account
Open browser → http://localhost:3000 Click "Sign up" and create a LOCAL admin account (this is only local — no internet account needed) Select a model from the dropdown at top (should list your Ollama models) Type a test message and verify you get a response
Step 4 — Create a custom system prompt (test Modelfile concept)
# In Open WebUI: click your username → Settings → System Prompt # Add: "You are an expert IAM engineer specialising in PingFederate, # CyberArk, and Saviynt. Answer questions concisely and technically." # Now every conversation starts with this context.
Step 5 — Stop and start commands for future use
docker stop open-webui # Stop it docker start open-webui # Start it again # The --restart always flag means it auto-starts on Windows boot
localhost:3000 loads the chat UI. You can select qwen3:8b from the model dropdown and chat with it. Responses stream in, not appearing all at once.
If Open WebUI cannot find your Ollama models: the container uses host.docker.internal to reach your Windows Ollama. Verify Ollama is running: ollama list in a separate terminal. Try restarting Docker Desktop if models don't appear.
If localhost:3000 shows a connection error: docker logs open-webui to see what went wrong. Most common cause: port 3000 is in use by another app. Change -p 3000:8080 to -p 3001:8080 and visit localhost:3001 instead.
4
Install VS Code + Continue.dev (AI Coding Assistant)
Continue.dev is a free GitHub Copilot replacement that uses your local Ollama models. Zero subscription. Works for code completion and inline chat. Install both before writing any AI code.
Step 1 — Install VS Code
Go to: https://code.visualstudio.com Download "User Installer" for Windows 64-bit Run the installer ✓ Check "Add to PATH" during installation (important!) ✓ Check "Add 'Open with Code' to context menu" (optional but useful)
Step 2 — Install the Continue extension
Open VS Code Press Ctrl+Shift+X to open Extensions panel Search for "Continue" Install "Continue - Codestral, Claude, and more" (by Continue) Wait for install to complete, then reload VS Code when prompted
Step 3 — Configure Continue to use your local Ollama
# Click the Continue icon in the left sidebar (looks like a play button) # Click the gear icon (⚙) → "Open config.json" # Replace the entire file with: { "models": [ { "title": "Qwen3 8B (Local)", "provider": "ollama", "model": "qwen3:8b", "apiBase": "http://localhost:11434" }, { "title": "Phi-4 Mini (Fast)", "provider": "ollama", "model": "phi4-mini", "apiBase": "http://localhost:11434" } ], "tabAutocompleteModel": { "title": "Phi-4 Mini Autocomplete", "provider": "ollama", "model": "phi4-mini", "apiBase": "http://localhost:11434" } }
Step 4 — Test it works
# Open any .ts or .py file in VS Code # Press Ctrl+I (inline chat) and type: "explain this code" # Or highlight some code, right-click → "Ask Continue" # You should see a response streaming from your local model
Ctrl+I in VS Code opens the Continue chat panel. You can ask questions about your code and get responses from your local Qwen3 8B model with no internet request going out.
Tab autocomplete can slow down VS Code if your CPU is busy. If VS Code feels laggy, disable autocomplete in Continue settings and only use the chat feature (Ctrl+I) when you need it.
5
Install Node.js + TypeScript (Phase 1–2 coding)
You will write your first AI tools in TypeScript since you already know it. Ollama's REST API is plain HTTP — no AI-specific library needed. This step verifies your Node.js setup is clean.
Step 1 — Install Node.js LTS
Go to: https://nodejs.org Download the LTS version (v20.x or v22.x) Run installer with all defaults Reopen terminal after install
Step 2 — Verify installation
node --version # Expected: v20.x.x or higher npm --version # Expected: 10.x.x or higher
Step 3 — Set up your first AI project
mkdir my-ai-tools && cd my-ai-tools npm init -y npm install typescript ts-node @types/node --save-dev npx tsc --init # Creates tsconfig.json. You are ready to write TypeScript.
Step 4 — Test calling Ollama from TypeScript
# Create file: test-ollama.ts import * as https from 'http'; const body = JSON.stringify({ model: 'qwen3:8b', messages: [{ role: 'user', content: 'Say hello in one word' }], stream: false }); const options = { hostname: 'localhost', port: 11434, path: '/api/chat', method: 'POST', headers: { 'Content-Type': 'application/json' } }; const req = https.request(options, (res) => { let data = ''; res.on('data', chunk => data += chunk); res.on('end', () => { const json = JSON.parse(data); console.log(json.message.content); }); }); req.write(body); req.end(); # Run it: npx ts-node test-ollama.ts # Expected: single word like "Hello!" from the model
TypeScript file runs without errors and prints a response from the model. You have made your first programmatic LLM call.
6
Install Python 3.11+ with Virtual Environments
Python is required for Phase 3 (LlamaIndex, ChromaDB) and Phase 4 (LangChain, MCP). Install it now so you have time to get comfortable before you need it. Virtual environments (venv) are mandatory — AI libraries conflict badly without isolation.
Step 1 — Install Python
Go to: https://www.python.org/downloads/ Download Python 3.11.x or 3.12.x (NOT 3.13 — some AI libs lag behind) Run installer ✓ CRITICAL: Check "Add Python to PATH" at the bottom of the first screen ✓ Choose "Install Now" (standard install is fine)
Step 2 — Verify Python and pip
python --version # Expected: Python 3.11.x or 3.12.x pip --version # Expected: pip 24.x.x # If "python" is not found, try "python3" instead # If neither works: reboot and try again (PATH issue)
Step 3 — Create your first virtual environment (do this for EVERY AI project)
mkdir my-rag-project && cd my-rag-project # Create the virtual environment python -m venv venv # Activate it (do this every time you open a new terminal for this project) venv\Scripts\activate # Windows CMD # OR .\venv\Scripts\Activate.ps1 # Windows PowerShell # You will see (venv) prefix on your prompt when it is active # Example: (venv) PS C:\Users\you\my-rag-project>
Step 4 — Install base Phase 2 Python packages (inside venv)
# Make sure (venv) is showing in your prompt first! pip install ollama requests python-dotenv pip freeze > requirements.txt # Save exact versions # These are enough for all Phase 2 Python work
Step 5 — Test calling Ollama from Python
# Create file: test_ollama.py import ollama response = ollama.chat( model='qwen3:8b', messages=[{'role': 'user', 'content': 'What is SAML in one sentence?'}] ) print(response['message']['content']) # Run it (with venv active): python test_ollama.py
Python test script runs and prints a one-sentence SAML explanation. The (venv) prefix shows in your terminal prompt.
Never install AI libraries globally (without venv). LlamaIndex, LangChain, and ChromaDB have conflicting dependencies. Always activate venv before any pip install. If you forget: delete the venv folder and recreate it.
If PowerShell blocks the Activate.ps1 script: run Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser in PowerShell as administrator. This is a Windows security policy for running scripts.

Scenario-by-Scenario Brutal Analysis
RAG Pipelines and AI Orchestration Python wins. Not close.
Python wins by a margin that cannot be argued with. LlamaIndex and LangChain are Python-first projects. Their JavaScript ports are maintained as afterthoughts — features land in Python weeks to months earlier, documentation is 10x richer, and Stack Overflow coverage overwhelmingly assumes Python. When you hit a bug in LangChain.js you will find two StackOverflow posts; in LangChain Python you will find thirty and a GitHub issue with a fix.

TypeScript's honest problem: The type system is beautiful but the AI library ecosystem has not caught up. You will spend time fighting outdated type definitions, missing methods, and "this feature is Python only" GitHub issues. Frustrating for someone who values type safety.

Java's honest problem: It is not just behind — it is practically absent. Langchain4j exists but it is a niche project with a small community. If you build RAG in Java, you are on your own for most real-world problems. The cost is not the language — it is the ecosystem vacuum.
Python → use itTypeScript → avoid for RAGJava → do not even start
Calling Ollama REST API for Quick Tools TypeScript wins for you specifically.
This is the scenario where TypeScript is genuinely the right call — and it aligns perfectly with your existing skills. The Ollama API is dead-simple HTTP. TypeScript gives you type safety on the request/response shapes, excellent IDE autocomplete, and you already know the ecosystem. Zero learning tax, maximum productivity in Phase 1–2.

Python is fine here too — the ollama pip SDK is clean. But there is no reason to learn Python for this task when you already have TypeScript. Use your existing skills to build momentum in Phases 1 and 2 before switching to Python for Phase 3 where it genuinely matters.

Node.js (plain JS) works but gives up type safety for no gain. There is no reason to use plain JS when you already know TypeScript.

Java here is bureaucracy for no reason. A Spring Boot service to call one HTTP endpoint is 200 lines of boilerplate for what TypeScript does in 15 lines. The JVM startup time alone makes it painful for scripting and iteration.
TypeScript → use it (you know it)Python → equally good, pick oneJava → overkill
Enterprise IAM System Integration Java wins — but only if the codebase already exists.
This is Java's genuine home turf. If you have an existing Java service that talks to PingFederate or CyberArk, extending it with an AI layer makes architectural sense. You already have the LDAP libraries, the SOAP clients, the authentication utilities. Adding a simple Ollama HTTP call to an existing Java service is five lines of code with OkHttp.

But if you are building from scratch for AI tooling, Java's advantage disappears. You are now paying 5x the boilerplate cost for no ecosystem benefit. A new IAM integration tool for AI purposes should be Python (if it needs RAG) or TypeScript (if it is a REST service).

Python for new integrations is often better than people expect. The requests library is excellent. PingFederate, Saviynt, and CyberArk all have REST APIs that Python handles with zero friction.

The brutal truth about Java and AI: Java's enterprise strength is stability and ecosystem — but the AI ecosystem is moving too fast for Java's conservative release cycle. A Java AI library that is 6 months old may already be obsolete. Python's faster iteration cycle is an asset in this space, not a liability.
Java → use if extending existing codePython → use for new AI integrationsTypeScript → good for REST wrappers
Production API Services and REST Wrappers TypeScript (NestJS) or Python (FastAPI) — genuinely tied.
FastAPI (Python) is exceptional for AI service wrapping. Automatic OpenAPI docs, async support, Pydantic validation, and it integrates seamlessly with LlamaIndex/LangChain since they are Python too. One language for your RAG logic and your API layer is a real productivity win. No context switching.

NestJS (TypeScript) is genuinely enterprise-grade and your TypeScript expertise means zero learning curve on the framework. If the service is primarily a stateless REST proxy over AI calls with no heavy data processing, NestJS is a valid choice and your team will maintain it more easily if they know TypeScript.

Spring Boot (Java) is not wrong for production services — it is battle-tested, and if your ops team already runs Java in production, it is a sensible choice for a production AI service that needs to live in your existing infrastructure. The startup time and memory overhead matter less in long-running services.

The pragmatic recommendation: If your AI service does significant data processing, embeddings, or RAG — use FastAPI, keep everything Python. If it is purely a proxy/wrapper with simple business logic — use TypeScript where your team has existing skills.
FastAPI → use when service does AI workNestJS → use when service is pure APISpring Boot → use in existing Java infrastructure
Quick Experimentation and Prototyping Python wins. By a country mile.
Python's REPL, Jupyter notebooks, and zero-friction scripting model are unmatched for AI experimentation. You run one line, see the result, iterate. When you are trying to figure out why your RAG pipeline returns irrelevant chunks, being able to run embedding = embed("test string") in a notebook cell and inspect the result is invaluable. You cannot replicate this efficiently in TypeScript or Java.

TypeScript's compile step — even with ts-node — adds friction to rapid experimentation. Not a problem for production code but genuinely annoying when you are in "figure out why this doesn't work" mode at 11pm.

Java is the worst option for experimentation by a wide margin. Even with a modern IDE, the build-run-debug cycle for a Java experiment that would take 2 minutes in Python takes 10. The boilerplate cost (main class, package declarations, import management) is pure overhead when all you want to do is test if an embedding is working.
Python → always for experimentsTypeScript → fine for finished toolsJava → never for AI experiments
The Honest Bottom Line for Your Specific Situation
Given your background — strong TypeScript/Java enterprise engineer, zero Python — here is the brutally honest learning path recommendation:

Phases 1–2: Stay in TypeScript. You will build faster, learn AI concepts without fighting a new language, and produce real tools. The Ollama REST API asks nothing of you that TypeScript cannot handle cleanly.

Phase 3 onwards: Switch to Python. Not because Python is inherently better than TypeScript — but because the RAG ecosystem lives there, the tutorials assume it, the debugging resources are there, and fighting the ecosystem to stay in TypeScript will cost you three times as long. This is a pragmatic decision, not an ideological one.

Java: Keep it for your existing enterprise integrations. If you build an MCP server that wraps a Java-based IAM system, it is entirely valid to have a Java service behind a Python MCP adapter layer. Use each language where your existing codebase and team already live in it.
Phase 1–2: TypeScript Phase 3–4: Python Java: existing integrations only

Common Mistakes

Expanded list. The ways experienced engineers specifically waste time entering AI engineering. Read all of these before starting Phase 1.

⚡
Starting with too large a model
Beginners install 13B on 32 GB RAM, get 2 tok/sec, give up. Start with Phi-4 Mini for speed, Qwen3 8B for quality. Both run at 10+ tok/sec and handle everything in this roadmap.
💸
Paying for cloud APIs in Phases 1–2
You need zero cloud API keys to complete Phases 1–3. Everything runs locally and free. Save credits for benchmarking your local system against a frontier model in Phase 4.
📚
Over-reading before building anything
AI engineering is learned by doing. After 30 minutes of reading, build something — even if it breaks. A broken script teaches 10x more than 3 hours of documentation.
🐍
Refusing to learn Python because you know Node/Java
All serious AI tooling is Python-first. One week of learning, massive payoff from Phase 3 onwards. You do not need to master it — just enough to read docs and run scripts.
🏗️
Installing LangChain before Phase 4
LangChain is complex and confusing for beginners. In Phases 1–3, raw Ollama API calls and LlamaIndex give a much clearer understanding of what is actually happening under the hood.
📐
Not version-pinning Python dependencies
AI libraries change fast. A tutorial from 6 months ago may break with the latest LlamaIndex. Always pin: pip install "llama-index==0.10.68" and commit requirements.txt on day one.
🎯
Building agents before mastering RAG
Agents are Phase 4. Master prompt engineering → structured output → RAG in order. Each builds on the previous. Skipping creates systems you cannot debug because you don't understand the components.
💾
Ignoring context window limits
Stuffing too much into a context window degrades quality sharply. Quality drops past 16k in practice even on 32k models. In RAG: retrieve 5–8 chunks, not 50. Less context, better answers.
🌡️
Never adjusting temperature and ignoring it
Temperature controls randomness. Default (~0.8) is good for chat. For structured JSON output or code, set temperature to 0.1–0.2 or you get unpredictably formatted responses that break your parser. In Ollama Modelfile: PARAMETER temperature 0.1. This single change dramatically improves reliability for tool-based tasks.
🧪
Building with fake/toy data, never testing on real data
RAG pipelines that work perfectly on your 5 hand-crafted test sentences collapse when you feed them 300-page vendor PDFs with tables, footnotes, and headers. Always test with your actual documents as early as week 2 of Phase 3. Real data exposes chunking and retrieval bugs that toy data cannot.
🖥️
Building a frontend before the AI pipeline works
A beautiful UI around a broken RAG pipeline is a broken RAG pipeline with lipstick. Build the backend pipeline first, test it via curl or simple Python scripts, validate accuracy, then add a UI. Every hour spent on frontend before the backend is solid is a wasted hour.
🐳
Over-engineering with Docker/Kubernetes too early
In Phases 1–3, run everything locally with direct Python scripts. Adding Docker for your RAG pipeline before it even works correctly adds debugging complexity with zero benefit. Containerise when your system works and you want to share or deploy it — not before.
📋
Not logging LLM inputs and outputs
When your agent gives a wrong answer at 11pm, you will want to know exactly what prompt it received and what the model said at each step. Add simple file-based logging to every LLM call from day one. Print the full prompt, the full response, and the tool calls. Without this you are debugging blind.
🧠
Assuming the model understands your domain without a system prompt
A base model asked about "PAM" will give you a mix of PingAccess, Privileged Access Management, and random acronyms. A system prompt that says "You are an IAM engineer. PAM means Privileged Access Management. PF means PingFederate." transforms response quality immediately. Write domain-specific system prompts for every tool you build.
📊
Shipping without any evaluation or measurement
The single biggest mistake in AI engineering. "It seemed to work when I tested it" is not evaluation. Build a small test set of 20 questions with known correct answers for every RAG system. Run it before and after any change. If you cannot measure accuracy, you cannot improve it — and you will not know when a model update or chunking change broke something.
🔄
Rebuilding the index on every code change
Rebuilding your ChromaDB vector index takes minutes to hours depending on document size. Beginners re-index everything during every test run. Persist your index to disk (ChromaDB supports this with PersistentClient) so it survives restarts. Only rebuild when documents actually change.
🎭
Treating every AI failure as a model quality problem
When your RAG system gives wrong answers, beginners immediately switch to a larger model. 80% of RAG failures are actually chunking strategy problems, retrieval quality issues, or bad system prompts — not model capability limits. Debug the pipeline before blaming the model. Switch the model last.
