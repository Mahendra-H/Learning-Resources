vLLM

High-throughput LLM inference — the speed layer
🎬 Origin Story — Why It Was Built

Woosuk Kwon at UC Berkeley noticed something alarming in 2023: GPU clusters serving LLMs were wasting 60–80% of their memory on the KV cache — the data structure LLMs use to remember context across tokens. The waste happened because existing systems pre-allocated worst-case memory per request, leaving most of it unused and fragmented. Kwon borrowed the paging concept from operating system memory management — the same idea that lets your laptop run more programs than it has RAM for. The result, PagedAttention, became the paper that changed production LLM serving. vLLM deployed PagedAttention as an open-source serving engine and immediately achieved 10–24× higher throughput than naive serving. Today every major LLM serving deployment — Mistral, Anyscale, AWS, Google — uses vLLM or the ideas from it.

Before vs After — The Problem It Solves

❌ Naive serving — sequential, GPU starved
# Sequential: users wait in line
# GPU sits at 20% utilization between requests
from transformers import pipeline

model = pipeline("text-generation",
  model="meta-llama/Llama-2-7b")

# User 1 → wait (GPU busy) → response
# User 2 → wait → response
# User 3 → wait → response
# 10 users = 10× single user latency

# Problems:
# - 1 request at a time
# - KV cache pre-allocated = GPU memory wasted
# - No batching = throughput of a single lane road
# - 100 req/min capacity on a $10k A100
✅ vLLM — batched, GPU saturated
from vllm import LLM, SamplingParams

llm = LLM(model="meta-llama/Llama-2-7b")
params = SamplingParams(
  temperature=0.7, max_tokens=200
)

# All users handled simultaneously
prompts = [
  "Summarize quantum computing",
  "Explain neural networks",
  "What is Rust programming?",
  # ...100 more
]

# vLLM batches automatically with PagedAttention
outputs = llm.generate(prompts, params)

# Benefits:
# - GPU at 80-90% utilization
# - 24× throughput vs naive
# - 1000+ req/min on same A100

Quick Start

bash + python · requires GPU for production, CPU for small models
# Install
pip install vllm

# Option 1: OpenAI-compatible API server (drop-in replacement)
python -m vllm.entrypoints.openai.api_server \
  --model "mistralai/Mistral-7B-Instruct-v0.3" \
  --port 8000

# Now use it exactly like OpenAI API — just change base_url
from openai import OpenAI
client = OpenAI(base_url="http://localhost:8000/v1", api_key="none")
response = client.chat.completions.create(
  model="mistralai/Mistral-7B-Instruct-v0.3",
  messages=[{"role":"user", "content":"Explain RAG in 2 sentences"}]
)
print(response.choices[0].message.content)

# Option 2: Python API (for batch processing)
from vllm import LLM, SamplingParams
llm     = LLM(model="facebook/opt-125m")  # small model, no GPU needed
params  = SamplingParams(temperature=0.7, max_tokens=100)
outputs = llm.generate(["Tell me about Python", "What is AI?"], params)
for o in outputs:
  print(o.outputs[0].text)

Real-World Scenario — Replacing a $50k/Month OpenAI Bill

💸
Series B AI Startup · 200k daily active users
Cutting LLM inference cost from $50,000/month to $3,200/month
🔴 The Problem

The startup was spending $50,000/month on OpenAI API calls for a summarization feature. At Series B scale, investors started asking hard questions about unit economics. The LLM cost per user was 4× what it needed to be for a viable margin. OpenAI's rate limits were also becoming a bottleneck at peak hours.

🟢 The vLLM Solution

Deployed vLLM serving Llama 3 8B on 2 reserved A100 GPUs ($1,600/month total). Replaced the OpenAI API call with a local endpoint using vLLM's OpenAI-compatible server — zero code change required except the base_url. vLLM's batching handled 3× peak traffic vs before on the same hardware.

Result: $50,000/month → $3,200/month (94% cost reduction). Latency improved from 800ms (OpenAI) to 340ms (local vLLM). No rate limits. The OpenAI-compatible API meant zero changes to existing LangChain code — just swapped the base_url.

Anti-Patterns

Anti-Pattern #1
🔴 Symptom

vLLM crashes immediately or refuses to load the model with a CUDA error.

🟡 Root Cause

Running vLLM on a CPU machine. vLLM requires a CUDA-capable GPU. It is not optional — the PagedAttention kernel only runs on GPU.

🟢 Fix

For CPU-only machines use Ollama or HuggingFace pipeline. For cloud, use a GPU instance (AWS g5, Lambda Labs, RunPod) or switch to Modal.

Anti-Pattern #2
🔴 Symptom

Throughput is no better than naive serving despite using vLLM.

🟡 Root Cause

Sending one request at a time and waiting for the response before sending the next. Not using vLLM's batching at all.

🟢 Fix

Send requests concurrently using asyncio or a load-testing tool. vLLM's value is continuous batching — it only shows up when multiple requests arrive simultaneously.

Anti-Pattern #3
🔴 Symptom

Model output quality dropped noticeably after switching to vLLM from OpenAI.

🟡 Root Cause

Using INT4 quantization to fit a 13B model on a small GPU. Aggressive quantization degrades quality significantly.

🟢 Fix

Use a smaller model at float16 instead of a larger model at INT4. A 7B model at float16 usually outperforms a 13B at INT4 and uses comparable VRAM.

Ollama

Docker for LLMs — run any model locally in one command
🎬 Origin Story — Why It Was Built

Jeffrey Morgan built Ollama in 2023 after a specific frustration: getting any open-source LLM running locally required downloading model weights in the right format, understanding quantization, writing a server, managing dependencies, and configuring CUDA — a process that took hours even for experienced engineers. His insight was a direct analogy with Docker: Docker made containers accessible by hiding all the complexity behind docker run. Ollama would do the same for LLMs with ollama run. The project hit 20,000 GitHub stars in the first week. It's now the single fastest way to get a local LLM running — from zero to chatting in under 2 minutes on any Mac or Linux machine.

Before vs After — The Problem It Solves

Real-World Scenario — Local Coding Assistant

👨‍💻
Solo Developer · Personal productivity · Fully offline
A Cursor-like coding assistant that works without internet and costs $0/month
🔴 The Problem

A developer was spending $30/month on GitHub Copilot and Cursor. More importantly, they worked on client code covered by strict NDAs — sending source code to cloud APIs was a compliance violation they couldn't risk. They needed AI code assistance that was local, fast, and free.

🟢 The Ollama Solution

Anti-Patterns

Anti-Pattern #1
🔴 Symptom

Ollama works for you locally, but fails when your colleague tries it or when deployed to a server.

🟡 Root Cause

Treating Ollama as a production server. It's designed for single-user local use. It has no authentication, no rate limiting, no concurrency controls.

🟢 Fix

For multi-user or production serving, use vLLM (GPU) or wrap Ollama with FastAPI and add auth. Ollama is the dev environment; vLLM is the production environment.

Anti-Pattern #2
🔴 Symptom

App responds slowly or computer becomes sluggish while Ollama runs. Fans spin at full speed.

🟡 Root Cause

Running a 7B or 13B model on a machine with less RAM than the model needs. The model spills to swap memory, which is 100× slower than RAM.

🟢 Fix

Check your RAM: 7B needs ~8GB free RAM, 13B needs ~16GB. If constrained, use a smaller model (phi3 3.8B is excellent quality for its size).

Anti-Pattern #3
🔴 Symptom

Responses are incoherent or repetitive even for simple questions on a capable machine.

🟡 Root Cause

Using Q2 or Q3 quantization. These extreme compressions cut model size massively but degrade quality below usable thresholds.

🟢 Fix

Stick to Q4_K_M (default) or Q5_K_M for better quality. Only drop to Q2/Q3 if you have no choice due to RAM constraints — and accept the quality tradeoff.

FastAPI

The service counter — expose your AI as a REST API
🎬 Origin Story — Why It Was Built

Sebastián Ramírez released FastAPI in 2018. He was building APIs with Flask and Django and kept hitting the same walls: Flask had no type hints, no async support, and documenting the API was manual work. Django was heavyweight for pure API services. He wanted three things simultaneously: fast to write, fast to run, and self-documenting. He built FastAPI on top of Starlette (async web framework) and Pydantic (data validation). The result: type-annotated Python functions become REST endpoints, validation is automatic, and you get a free interactive Swagger UI at /docs with zero extra work. For AI APIs in particular — where you need async for LLM calls, validation for prompt payloads, and streaming for token output — FastAPI is close to the ideal framework.

Before vs After — The Problem It Solves

❌ Flask — sync, manual validation, no docs
from flask import Flask, request, jsonify

app = Flask(__name__)

# No type hints — manual validation everywhere
# No async — LLM call blocks the whole server
@app.route('/ask', methods=['POST'])
def ask():
  data = request.get_json()

  # Manually validate — easy to miss edge cases
  if 'question' not in data:
    return jsonify({'error': 'Missing question'}), 400
  if len(data['question']) > 1000:
    return jsonify({'error': 'Too long'}), 400

  # Blocking call — no other requests handled while waiting
  answer = llm.invoke(data['question'])
  return jsonify({'answer': answer})
# No /docs UI · No streaming · No type safety
✅ FastAPI — async, auto-validation, auto-docs
from fastapi import FastAPI
from pydantic import BaseModel, Field
from fastapi.responses import StreamingResponse

app = FastAPI(title="My AI API")

# Pydantic validates automatically — no manual checks
class AskRequest(BaseModel):
  question: str = Field(max_length=1000)
  style:    str = "concise"

# Async — server handles other requests while LLM runs
@app.post("/ask")
async def ask(req: AskRequest):
  answer = await llm.ainvoke(req.question)
  return {"answer": answer.content}

# Streaming endpoint — tokens sent as they generate
@app.post("/stream")
async def stream(req: AskRequest):
  return StreamingResponse(
    llm.astream(req.question), media_type="text/plain"
  )
# Free /docs Swagger UI · Type safety · Async

Real-World Scenario — Team AI API Platform

🔌
40-person Tech Company · Cross-team AI access
A single AI API endpoint serving 5 different product teams
🔴 The Problem

Five product teams all needed LLM access. Each team was managing their own OpenAI keys, writing their own retry logic, paying separately, and building the same rate-limiting code from scratch. No usage visibility. No shared cost management. No consistent response format.

🟢 The FastAPI Solution

One FastAPI service exposing /summarize, /extract, /classify, /chat — all backed by the same LLM pool with shared rate limiting via a Redis token bucket. Pydantic schemas enforce consistent request/response formats. Auto-generated /docs replaced all internal API documentation. Usage logged per team per endpoint.

Anti-Patterns

Anti-Pattern #1
🔴 Symptom

Server appears to hang under moderate load. Single LLM call blocks all other incoming requests for 2–5 seconds.

🟡 Root Cause

Using llm.invoke() (synchronous) inside an async def endpoint. This blocks the entire async event loop.

🟢 Fix

Use await llm.ainvoke() for async LLM calls. Every LangChain LLM has an async version prefixed with a: ainvoke, astream, abatch.

Anti-Pattern #2
🔴 Symptom

Bad request payloads crash your handler with confusing Python errors instead of clean 422 responses.

🟡 Root Cause

Using request.json() and manually accessing dict keys without defining a Pydantic model. Missing the automatic validation FastAPI provides.

🟢 Fix

Always define a Pydantic BaseModel for every endpoint's request body. FastAPI returns a detailed 422 error automatically for invalid payloads — no extra code.

Anti-Pattern #3
🔴 Symptom

LLM exceptions bubble up to the user as unformatted 500 errors with Python stack traces in the response body.

🟡 Root Cause

No exception handling in endpoint handlers. LLM APIs can fail with rate limits, context length errors, content policy violations — all unhandled.

🟢 Fix

Wrap LLM calls in try/except. Return HTTPException(status_code=503, detail="LLM unavailable") for recoverable errors. Log the full trace server-side.

LangServe

Deploy any LangChain chain as an API in one line
🎬 Origin Story — Why It Was Built

After LangChain gained adoption, the LangChain team noticed a pattern: every developer deploying a chain was writing the same FastAPI boilerplate. Define a request model. Add an invoke endpoint. Add a batch endpoint. Add a streaming endpoint. Handle errors. Repeat for every new chain. LangServe, released in late 2023, collapsed that boilerplate into a single function call: add_routes(app, chain, path="/my-chain"). Immediately, every chain gets four free endpoints — /invoke, /batch, /stream, and /playground — the last of which is a visual, browser-based UI to test your chain without any frontend code. For teams sharing LangChain work internally, LangServe went from "deploy in 2 days" to "deploy in 10 minutes."

Before vs After — The Problem It Solves

❌ Manual FastAPI for a LangChain chain
from fastapi import FastAPI
from pydantic import BaseModel
from fastapi.responses import StreamingResponse

app = FastAPI()
chain = prompt | llm | parser  # your chain

class Input(BaseModel):
  input: dict

# Manually write all 4 endpoints
@app.post("/invoke")
async def invoke(body: Input):
  return await chain.ainvoke(body.input)

@app.post("/batch")
async def batch(body: Input):
  return await chain.abatch(body.input)

@app.post("/stream")
async def stream(body: Input):
  return StreamingResponse(chain.astream(body.input))

# No playground UI
# Same boilerplate for every chain you deploy
✅ LangServe — one line, 4 endpoints + playground
from fastapi import FastAPI
from langserve import add_routes

app   = FastAPI(title="My Chain API")
chain = prompt | llm | parser  # same chain

# One line → 4 endpoints automatically
add_routes(app, chain, path="/ask")

# You now get FOR FREE:
# POST /ask/invoke   → single call
# POST /ask/batch    → parallel calls
# POST /ask/stream   → streaming response
# GET  /ask/playground → visual UI to test!

# Add multiple chains just as easily
add_routes(app, summary_chain, path="/summarize")
add_routes(app, translate_chain, path="/translate")

# Run: uvicorn main:app --reload
# Visit: localhost:8000/ask/playground
🏢 In the Wild

Teams at companies like Replit and Weights & Biases use LangServe to share internal LangChain chains across teams as self-documenting APIs. The /playground feature is especially popular for non-engineers — product managers and designers can test and iterate on prompt behavior without touching any code.

Anti-Patterns

Anti-Pattern #1
🔴 Symptom

You're trying to deploy regular Python functions or non-LangChain ML models with LangServe and getting confusing errors.

🟡 Root Cause

LangServe only works with LangChain Runnables (chains, agents, etc.). It does not accept arbitrary functions.

🟢 Fix

Use RunnableLambda to wrap any Python function as a LangChain Runnable: from langchain.schema.runnable import RunnableLambda.

Anti-Pattern #2
🔴 Symptom

Team members don't know how to test the deployed chain. You're writing curl documentation nobody reads.

🟡 Root Cause

Not telling anyone about the /playground endpoint. It's the most valuable feature and consistently overlooked.

🟢 Fix

Share the /playground URL as the primary way to interact with your chain. It's a visual interface anyone can use — no curl, no Postman, no code.

Anti-Pattern #3
🔴 Symptom

Frontend shows partial responses or garbled data when using the /stream endpoint.

🟡 Root Cause

Reading the stream as plain text. LangServe streams Server-Sent Events (SSE), which have a specific format that needs to be parsed.

🟢 Fix

Use the official LangServe JS/TS client: npm install langserve. It handles SSE parsing automatically and provides a RemoteRunnable client.

Modal

Serverless GPU — run Python functions in the cloud, pay per second
🎬 Origin Story — Why It Was Built

Erik Bernhardsson — former CTO of Better.com and before that the creator of Spotify's recommendation system — founded Modal in 2021. His thesis came from years of pain: running ML models in production required maintaining Kubernetes clusters, Docker registries, GPU drivers, and deployment pipelines that took teams weeks to set up and months to maintain. GPU compute should be as easy as calling a Python function. You decorate your function with @app.function(gpu="A10G"), run it, and Modal provisions the GPU, installs dependencies in an isolated container, runs your code, and tears the container down — all in seconds. No Docker. No Kubernetes. No servers to maintain. Pay only for the compute seconds you actually use.

Before vs After — The Problem It Solves

❌ Before Modal — DevOps nightmare
# Running GPU inference before Modal:

# 1. Launch EC2 GPU instance (p3.2xlarge = $3.06/hr)
#    — pays even when idle

# 2. Write Dockerfile
# FROM nvidia/cuda:12.0-base
# RUN pip install torch transformers ...
# COPY model_server.py .
# CMD ["python", "model_server.py"]

# 3. Build and push to ECR
# docker build -t my-model .
# docker push 123456.ecr.us-east-1.amazonaws.com/my-model

# 4. Write CloudFormation / Terraform
# ... 200 lines of YAML ...

# 5. Debug GPU driver issues for 3 hours

# 6. Pay $2,200/month for the instance
# whether you use it or not
✅ Modal — decorate and run
import modal

app = modal.App("my-inference")

# This entire function runs on a cloud GPU
# Modal handles: container, GPU, dependencies
@app.function(
  gpu="A10G",                       # 24GB VRAM
  image=modal.Image.debian_slim().pip_install(
    "transformers", "torch"
  ),
  timeout=300
)
def run_inference(prompt: str) -> str:
  from transformers import pipeline
  pipe = pipeline("text-generation",
    model="meta-llama/Llama-3-8B-Instruct")
  return pipe(prompt, max_new_tokens=200)[0]['generated_text']

@app.local_entrypoint()
def main():
  result = run_inference.remote("Explain vLLM")
  print(result)
# modal run inference.py — that's it
# Pay: ~$0.0006 per second of GPU time

Anti-Patterns

Anti-Pattern #1
🔴 Symptom

Users complain of 3–5 second delays on every request to your Modal-powered endpoint.

🟡 Root Cause

Cold starts on every request because the container is idle between calls and gets terminated after the grace period.

🟢 Fix

Add keep_warm=1 to your @app.function decorator. This keeps one container always alive, eliminating cold starts for the cost of one idle container.

Anti-Pattern #2
🔴 Symptom

Data written to local files inside Modal functions disappears between invocations.

🟡 Root Cause

Modal containers are ephemeral and stateless. Local file writes don't persist between function calls — the container is destroyed after each run.

🟢 Fix

Use Modal Volumes for persistent storage, or write to S3/GCS. Modal Volumes mount as a filesystem but persist across invocations: modal.Volume.

Anti-Pattern #3
🔴 Symptom

Modal costs are higher than expected. Bill shows hundreds of GPU hours even for a lightly-used endpoint.

🟡 Root Cause

Using keep_warm on a GPU function with high concurrency_limit. Each warm container bills for idle GPU time whether or not it's handling requests.

🟢 Fix

Only use keep_warm for latency-critical endpoints. For batch jobs, let containers cold start — the 3s overhead is fine when jobs run for minutes. Use CPU instances for health checks.

Replicate

Pre-deployed community models as instant API calls
🎬 Origin Story — Why It Was Built

Ben Firshman — a co-creator of Docker — founded Replicate in 2019. He saw the same problem in ML that Docker had solved in software: sharing a working environment was nearly impossible. A researcher would publish a model on arXiv and GitHub, but getting it running required cloning the repo, installing conflicting dependencies, configuring GPU drivers, and hunting down missing weights. Firshman containerized the entire problem: Replicate packages any model (code + weights + runtime) as an immutable, versioned container accessible through a single API call. No setup. No dependencies. Just call the API with your input, get the output back. Today Replicate hosts thousands of models — Stable Diffusion, Llama, Whisper, DALL-E alternatives — and is the fastest way for non-ML teams to consume AI capabilities.

Anti-Patterns

Anti-Pattern #1
🔴 Symptom

Monthly bill is shocking. Replicate costs are 10× what you expected for a moderately trafficked feature.

🟡 Root Cause

Using Replicate for high-volume production instead of prototyping. Per-call pricing is great for low volume, expensive at scale.

🟢 Fix

Break-even calculation: if you're spending >$150/month on Replicate for a specific model, run it on Modal or vLLM instead. Replicate is the exploration phase; self-hosted is the production phase.

Anti-Pattern #2
🔴 Symptom

Requests to less popular models time out or take 30+ seconds to return the first token.

🟡 Root Cause

Models with low request volume on Replicate are completely unloaded when idle. Cold starts for large models (13B+) can take 30–60 seconds.

🟢 Fix

Check the model's "recent runs" on the Replicate page. Popular models stay warm. For cold models, set client timeout to 90s and show a loading state. Or switch to a more popular equivalent.

Anti-Pattern #3
🔴 Symptom

Compliance team blocks the project. Legal says you can't send that type of data to external APIs.

🟡 Root Cause

Sending PII, medical records, legal documents, or proprietary business data to Replicate's cloud without checking data residency and privacy policies first.

🟢 Fix

For sensitive data, use Modal (your own cloud account, your data) or self-hosted Ollama/vLLM. Replicate is public cloud infrastructure — treat it like any third-party API.

python · server.py — run with: uvicorn server:app --reload
from fastapi import FastAPI, HTTPException
from fastapi.responses import StreamingResponse
from pydantic import BaseModel, Field
from langchain_community.llms import Ollama
from langchain.prompts import ChatPromptTemplate
from langchain.schema.output_parser import StrOutputParser
import time, logging

logging.basicConfig(level=logging.INFO)
app = FastAPI(title="Local AI API", version="1.0")

# ── LLM + Chain setup (once at startup) ──────────────────────
llm    = Ollama(model="llama3.2")       # Ollama must be running
parser = StrOutputParser()

# ── Request / Response models ─────────────────────────────────
class ChatRequest(BaseModel):
  message:     str = Field(min_length=1, max_length=2000)
  system:      str = "You are a helpful AI assistant."
  temperature: float = Field(default=0.7, ge=0.0, le=1.0)

# ── Endpoints ─────────────────────────────────────────────────
@app.get("/health")
async def health():
  return {"status":"ok", "model":"llama3.2", "backend":"ollama"}

@app.post("/chat")
async def chat(req: ChatRequest):
  prompt = ChatPromptTemplate.from_messages([
    ("system", req.system),
    ("human", "{message}")
  ])
  chain = prompt | llm | parser
  try:
    t0  = time.time()
    ans = await chain.ainvoke({"message": req.message})
    logging.info(f"chat | {time.time()-t0:.2f}s | {len(ans)} chars")
    return {"response": ans, "model":"llama3.2"}
  except Exception as e:
    raise HTTPException(503, str(e))

@app.post("/chat/stream")
async def chat_stream(req: ChatRequest):
  prompt = ChatPromptTemplate.from_messages([
    ("system", req.system),
    ("human", "{message}")
  ])
  chain = prompt | llm | parser
  async def generate():
    async for chunk in chain.astream({"message": req.message}):
      yield chunk
  return StreamingResponse(generate(), media_type="text/plain")

#   -d '{"message": "What is a vector database?"}'

@modal.asgi_app()
def fastapi_app():
  from fastapi import FastAPI
  from vllm import LLM, SamplingParams

  app = FastAPI()
  llm = LLM(model="meta-llama/Llama-3-8B-Instruct")

  @app.post("/generate")
  async def generate(prompt: str):
    params  = SamplingParams(temperature=0.7, max_tokens=200)
    outputs = llm.generate([prompt], params)
    return {"text": outputs[0].outputs[0].text}

  return app
