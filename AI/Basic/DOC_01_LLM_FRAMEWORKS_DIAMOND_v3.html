LangChain

The assembly line for LLM logic
🎬 Origin Story — Why It Was Built

Harrison Chase was a machine learning engineer at Robust Intelligence in late 2022. He kept writing the same boilerplate across every project: format a prompt string, call the OpenAI API, extract the content, parse the output, pass it to the next prompt, repeat. Every AI feature was 80% identical plumbing and 20% actual logic. He open-sourced LangChain in October 2022 as a way to standardize that plumbing. The community response was immediate — 50,000 GitHub stars in 6 months — because every ML engineer had the same frustration. The insight: if everyone is building the same connective tissue, that connective tissue should be a library, not copied code.

Before vs After — The Problem It Solves

❌ Without LangChain — Raw approach
# Every project: same boilerplate, copy-pasted
import openai, json

def summarize_then_title(text):
  client = openai.OpenAI()

  # Step 1: manually format prompt
  p1 = f"Summarize this text:\n{text}"
  r1 = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role":"user","content":p1}]
  )
  # Step 2: manually extract content
  summary = r1.choices[0].message.content

  # Step 3: manually build next prompt
  p2 = f"Write a title for:\n{summary}"
  r2 = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role":"user","content":p2}]
  )
  # Step 4: manually extract again
  title = r2.choices[0].message.content

  # Do this 10 more times for a real app...
  return {"summary": summary, "title": title}
✅ With LangChain — Clean, composable
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate
from langchain.schema.output_parser import StrOutputParser

llm    = ChatOpenAI(model="gpt-4o-mini")
parser = StrOutputParser()

# Declare steps as reusable components
summarize = ChatPromptTemplate.from_template(
  "Summarize this:\n{text}") | llm | parser

get_title = ChatPromptTemplate.from_template(
  "Write a title for:\n{summary}") | llm | parser

# Chain them — output of one flows to next
pipeline = (
  summarize
  | (lambda s: {"summary": s})
  | get_title
)

title = pipeline.invoke({"text": my_text})

The LCEL Pipe — The Core Mental Model

The | pipe is the single most important thing in LangChain. It chains steps together — each step's output becomes the next step's input, exactly like Unix shell pipes.

📐 Diagram 2 — LCEL Pipeline: What Type Flows Between Each Step
⚡ Gotcha — | Is Not Bitwise OR

LangChain overrides Python's __or__ method on all its objects. So prompt | llm returns a runnable chain, not a number. That's why it looks like magic — and exactly why it works.

Real-World Scenario — Sales Follow-Up Automation

🏢
B2B SaaS · Mid-size Company · 40-person sales team
Turning call transcripts into personalized follow-up emails
🔴 The Problem

Sales reps were spending 90 minutes after every customer call manually writing follow-up emails. The emails were inconsistent — different reps emphasized different points. Deal velocity suffered. Managers had no way to audit quality at scale.

🟢 The LangChain Solution

A 4-step LCEL pipeline: (1) Extract key topics from transcript with PydanticOutputParser, (2) look up customer tier from CRM via a @tool, (3) generate email draft matching tone to tier, (4) score draft quality and regenerate if score < 7. Single chain.invoke() call per transcript.

Result: 90-minute task → 8 seconds. Email quality scores increased 34%. LangChain was the right tool because it needed chaining + structured output + tool use + quality check — exactly what LCEL was built for.

Anti-Patterns — The Mistakes Everyone Makes

Anti-Pattern #1
🔴 Symptom

Chain is slow and expensive. Every request takes 3–5 seconds even for simple tasks.

🟡 Root Cause

Re-creating the ChatOpenAI() and chain objects inside the request handler on every single call.

🟢 Fix

Instantiate the LLM and chain once at module level. Reuse the same object for all requests. Cold-start cost paid once.

Anti-Pattern #2
🔴 Symptom

App crashes unpredictably when the LLM returns output in a slightly different format than expected.

🟡 Root Cause

Using StrOutputParser and then manually parsing the string with split() or regex. Fragile against any variation.

🟢 Fix

Use PydanticOutputParser with a typed model. The LLM is told exactly what JSON to return and Pydantic validates it.

Anti-Pattern #3
🔴 Symptom

Conversation doesn't remember previous messages even though you added Memory to the code.

🟡 Root Cause

Creating a new ConversationBufferMemory() instance on every API request. Each call starts fresh.

🟢 Fix

Store the memory object in a session store (Redis, dict keyed by user ID) and pass the same instance on every turn for that user.

🏢 In the Wild

ChatGPT's Code Interpreter is an LLM + Python execution tool, chained together. Perplexity is an LLM + web search tool + citation parser — a chain. Every AI product that "does something beyond text" uses LangChain's exact tool-calling pattern.

LangGraph

When straight lines aren't enough
🎬 Origin Story — Why It Was Built

The LangChain team released their agent executor in 2023. Developers loved it — until they hit the wall: the agent was a black box. You couldn't inspect state mid-run, you couldn't add a loop-back on failure, and conditional routing was a hack. Enterprise teams refused to ship it to production because they couldn't explain what the agent was doing. In late 2023, the team rebuilt the agent layer from scratch as an explicit, inspectable graph. Every node is visible. Every edge is declared. Every state transition is logged. That transparency is why Fortune 500 companies adopted LangGraph when they wouldn't touch the original agent executor.

Before vs After — The Problem It Solves

❌ Without LangGraph — Spaghetti while loop
# Manually managing a loop — fragile and unreadable
state = {"question": q, "answer": "", "attempts": 0}

while True:
  # Manual answer generation
  state["answer"] = llm.invoke(state["question"]).content
  state["attempts"] += 1

  # Manual quality check
  score = llm.invoke(
    f"Rate 1-10: {state['answer']}"
  ).content

  # Manual routing — grows unmanageable
  if "8" in score or "9" in score:
    break
  elif state["attempts"] >= 3:
    break
  # No visibility, no checkpointing,
  # no way to add a new branch without
  # refactoring the entire function
✅ With LangGraph — Explicit, inspectable graph
from langgraph.graph import StateGraph, END

# State is typed and explicit
class State(TypedDict):
  question: str; answer: str; attempts: int

# Each node is a pure function
def generate(s): ...   # returns updated state
def check(s):    ...   # returns updated state

# Routing is explicit and testable
def route(s) -> str:
  if s["score"] >= 8 or s["attempts"] >= 3:
    return "done"
  return "retry"

# Graph is declared, not implied
g = StateGraph(State)
g.add_node("generate", generate)
g.add_node("check", check)
g.add_conditional_edges("check", route,
  {"done": END, "retry": "generate"})
# Inspectable · Testable · Production-safe

Real-World Scenario — AI Code Review Platform

🔧
Dev Tools SaaS · 15-person engineering team
An agent that reviews PRs, fixes bugs, and loops until tests pass
🔴 The Problem

Code review was inconsistent. Senior engineers applied different standards. Junior devs had no automated feedback loop. The team wanted an AI reviewer that didn't just comment — but actually fixed issues and verified its own fixes ran correctly.

🟢 The LangGraph Solution

4-node graph: (1) Read PR diff, (2) Identify issues, (3) Generate fix, (4) Run tests with exec(). Conditional edge: if tests pass → done. If tests fail and attempts < 5 → loop back to node 3 with the error message added to state. The loop is the feature.

Result: 73% of simple bugs auto-fixed before human review. Review cycle time cut from 2.5 days to 18 hours. LangGraph was essential because the fix-verify loop couldn't be expressed as a straight chain.

Anti-Patterns — The Mistakes Everyone Makes

Anti-Pattern #1
🔴 Symptom

Graph runs forever and never reaches END. CPU spikes, app hangs, never returns a result.

🟡 Root Cause

Routing function's "good enough" condition is never satisfied — the LLM keeps scoring low but there's no max-attempts safety valve.

🟢 Fix

Always guard: if score >= 7 or attempts >= 3: return "done". Never let correctness be the only exit condition.

Anti-Pattern #2
🔴 Symptom

State from a previous run bleeds into the next. Outputs are wrong and non-deterministic between calls.

🟡 Root Cause

Mutating state in-place inside nodes: state["key"] = value. This modifies the shared dict instead of returning a new one.

🟢 Fix

Always return a new dict: return {**state, "key": new_value}. Spread and override — never mutate in place.

Anti-Pattern #3
🔴 Symptom

Agent fails in production. You can't tell which node ran, what state it was in, or where it went wrong.

🟡 Root Cause

Running graph.compile() with no debug logging. State transitions are invisible in production.

🟢 Fix

During development use graph.compile(debug=True). In production, log state at the start of each node function.

LlamaIndex

Give your LLM access to your own data
🎬 Origin Story — Why It Was Built

Jerry Liu was a software engineer at Uber in 2022, building an internal tool that needed to answer questions about Uber's internal documentation. He tried LangChain's document loaders and found the retrieval layer too thin — no real control over chunking strategy, no hybrid search, no smart handling of structured data. He built GPT Index (later renamed LlamaIndex) as a specialized data framework that treats ingestion and retrieval as first-class problems, not afterthoughts. The name changed because it was never about GPT specifically — it's about any LLM intelligently querying any data source. Today it's the default tool for anything involving "chat with your documents."

Before vs After — The Problem It Solves

❌ Without LlamaIndex — Manual RAG chaos
# Manual RAG — every team reinvents this
import PyPDF2, openai, numpy as np

# Step 1: manually read PDF
reader = PyPDF2.PdfReader("doc.pdf")
text = "".join([p.extract_text() for p in reader.pages])

# Step 2: manually chunk (naive — cuts sentences)
chunks = [text[i:i+512] for i in range(0,len(text),512)]

# Step 3: manually embed each chunk
client = openai.OpenAI()
embeds = []
for chunk in chunks:
  r = client.embeddings.create(
    model="text-embedding-ada-002", input=chunk)
  embeds.append(r.data[0].embedding)

# Step 4: manual cosine similarity search
def search(q, k=3):
  q_emb = client.embeddings.create(...).data[0].embedding
  scores = [np.dot(q_emb, e) for e in embeds]
  top = sorted(enumerate(scores),key=lambda x:-x[1])[:k]
  return [chunks[i] for i,_ in top]
# 50+ lines, no persistence, breaks on edge cases
✅ With LlamaIndex — 5 lines, production-ready
from llama_index.core import (
  VectorStoreIndex,
  SimpleDirectoryReader
)

# Load ALL file types from folder
documents = SimpleDirectoryReader(
  "./docs"
).load_data()

# Index = chunk + embed + store (automatic)
index = VectorStoreIndex.from_documents(
  documents
)

# Query with semantic search + LLM answer
engine = index.as_query_engine(
  similarity_top_k=3
)
response = engine.query(
  "What is the refund policy?"
)
print(response.response)
print(response.source_nodes)  # citations!
# Handles PDFs, MD, txt, CSV, web pages...

Real-World Scenario — Law Firm Contract Analysis

⚖️
Mid-size Law Firm · 120 Lawyers · 10,000 Contracts
Turning a 4-hour manual research task into an 8-minute query
🔴 The Problem

Junior lawyers spent 3–4 hours per case manually searching through 10,000 historical contracts to find precedents and relevant clauses. Searching was keyword-based, slow, and inconsistent. Senior partner review of every search added another day per case.

🟢 The LlamaIndex Solution

Index all 10,000 contracts once with LlamaIndex. Lawyers ask natural language questions. System retrieves top 5 most semantically relevant clauses across all contracts, cites the exact document and page number, and generates a summary answer. Entire search: 8 minutes.

Result: Research time 4 hours → 8 minutes (97% reduction). Citation accuracy 100% (LlamaIndex returns exact source nodes). The index rebuild runs every Sunday night when new contracts are filed. LlamaIndex was the only choice — nothing else handles 10,000 documents with source citation this cleanly.

Anti-Patterns — The Mistakes Everyone Makes

Anti-Pattern #1
🔴 Symptom

The wrong documents keep getting retrieved. Answers are off-topic or only partially relevant.

🟡 Root Cause

Chunk size too large. Each chunk contains multiple unrelated topics so the similarity score is diluted across all of them.

🟢 Fix

Start with chunk_size=512 and test 256 and 1024 on your specific data. The right size depends on how dense your content is.

Anti-Pattern #2
🔴 Symptom

App takes 30+ seconds to start and burns through your OpenAI embedding budget on every launch.

🟡 Root Cause

Re-indexing all documents from scratch on every application start instead of persisting the index to disk.

🟢 Fix

Call index.storage_context.persist("./storage") after first build. Then use load_index_from_storage() on every subsequent run.

Anti-Pattern #3
🔴 Symptom

Answer says "I don't know" or "not in context" even though you can clearly see the answer in your documents.

🟡 Root Cause

Either similarity_top_k=2 isn't retrieving enough context, or you used a different embedding model for indexing vs querying.

🟢 Fix

Increase similarity_top_k to 5. Verify both index time and query time use exactly the same embedding model name and version.

Hugging Face

The open-source model universe
🎬 Origin Story — Why It Was Built

Hugging Face started in 2016 as a teen chatbot app. Yes, really. By 2019, after releasing the Transformers library and opening the model hub to the public, founders Clément Delangue and Julien Chaumond made a decisive pivot: become the "GitHub of machine learning." Their thesis was simple but powerful — AI research was being bottlenecked because every lab was rebuilding the same model loading, tokenization, and inference code from scratch. The Transformers library standardized all of it behind one interface: from_pretrained() works for any model, any architecture. Today with 500,000+ models and the dominant embedding + fine-tuning ecosystem, it's the mandatory starting point for any open-source AI work.

Before vs After — The Problem It Solves

❌ Without Hugging Face — Custom everything
# Without HF: write model loading from scratch
import torch
from transformers_custom import BertModel  # your own fork

# Manually handle architecture differences
if model_type == "bert":
  config = load_bert_config(config_path)
  model = BertModel(config)
  state_dict = torch.load(weights_path)
  model.load_state_dict(state_dict)
elif model_type == "gpt2":
  # completely different loading logic...
  ...

# Manually handle tokenization per model
tokens = manual_tokenize(text, vocab_file,
  max_length=512, add_special_tokens=True)

# Manually move to GPU if available
device = "cuda" if torch.cuda.is_available() else "cpu"
model = model.to(device)

# Every new model = rewrite everything
✅ With Hugging Face — Consistent interface

# Same API for ANY task, ANY model

# Summarization

  model="facebook/bart-large-cnn")

# Sentiment (different model, same API)

result = classifier("I love this!")

# Local LLM generation (no API key!)
generator = pipeline("text-generation",
  model="TinyLlama/TinyLlama-1.1B-Chat-v1.0")
result = generator(prompt, max_new_tokens=100)

# Translation, NER, Q&A, image classification...
# All use the exact same pattern above.

Real-World Scenario — HR Resume Screener (On-Premise)

👥
Large HR Tech Company · 50,000 employees · GDPR-regulated
Screening 10,000 resumes per month without sending a single CV to the cloud
🔴 The Problem

Resume screening took 3 weeks and was deeply inconsistent. GDPR and internal data policy made it legally impossible to send CV data to OpenAI or any external API. The team was stuck between needing AI-quality screening and absolute data sovereignty requirements.

🟢 The Hugging Face Solution

Fine-tuned a BERT classifier on 2,000 manually-labeled resumes using Hugging Face Trainer. Model runs entirely on-premise on a single A100 GPU. Classifies each resume into 8 seniority/role categories in under 2 seconds. Zero data leaves the building. Model retrained monthly as hiring criteria evolve.

Result: 3-week screening process → 4 hours. 100% data compliance. $0 per-token cost. Hugging Face was the only viable option — no cloud API could satisfy the data residency requirements.

Anti-Patterns — The Mistakes Everyone Makes

Anti-Pattern #1
🔴 Symptom

Script hangs for 10 minutes on startup or times out. First run always fails.

🟡 Root Cause

Large model downloading on every run because you never set a persistent cache location. 7B model = ~14GB downloaded fresh each time.

🟢 Fix

Pre-download: huggingface-cli download model-name. Set HF_HOME=/your/persistent/path so the cache survives restarts.

Anti-Pattern #2
🔴 Symptom

CUDA out of memory error even though you have a GPU. Model loading crashes.

🟡 Root Cause

Loading model in full float32 precision. A 7B model at float32 needs ~28GB VRAM. Most GPUs only have 8–16GB.

🟢 Fix

Load with torch_dtype=torch.float16. Cuts VRAM requirement in half with less than 1% quality loss for most tasks.

Anti-Pattern #3
🔴 Symptom

Pipeline output looks completely wrong — garbled text, wrong labels, nonsense numbers.

🟡 Root Cause

Using the wrong pipeline task string for the model. A summarization model used for text-generation produces garbage. A classification model used for generation crashes.

🟢 Fix

Always check the model card at huggingface.co/model-name. The correct pipeline task and example code are listed on every model page.

🏢 In the Wild

Stability AI releases Stable Diffusion weights on Hugging Face. Meta releases Llama through it. Google releases Gemma through it. The model hub is now the de-facto distribution channel for every major open-source AI release — if a company open-sources a model, it goes to Hugging Face first.

Module 1 Capstone Build

One system, all 4 tools, fully working
🎯 What You're Building

Self-Improving Q&A System: User asks a question → LlamaIndex retrieves relevant docs → LangChain generates an answer → LangGraph checks quality → if score < 6, rephrases and retries (max 3 attempts) → returns answer + sources + attempt count. Four tools, one clean system.

python · capstone_qa.py — paste into a file and run
from typing import TypedDict, List
from langgraph.graph import StateGraph, END
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate
from langchain.schema.output_parser import StrOutputParser

# ── Setup (runs once at startup) ─────────────────────────────
docs   = SimpleDirectoryReader("./my_docs").load_data()
index  = VectorStoreIndex.from_documents(docs)
engine = index.as_query_engine(similarity_top_k=3)
llm    = ChatOpenAI(model="gpt-4o-mini")       # create once, reuse
parser = StrOutputParser()

# ── Typed State ──────────────────────────────────────────────
class QAState(TypedDict):
  question: str;  context: str;  sources: List[str]
  answer:   str;  quality: int;  attempts: int

# ── Nodes ────────────────────────────────────────────────────
def retrieve(s):
  r = engine.query(s["question"])
  srcs = [n.metadata.get("file_name","?") for n in r.source_nodes]
  return {**s, "context": r.response, "sources": list(set(srcs))}

def generate(s):
  chain = ChatPromptTemplate.from_template(
    "Context:\n{ctx}\n\nAnswer using only the context above: {q}"
  ) | llm | parser
  return {**s, "answer": chain.invoke({"ctx":s["context"],"q":s["question"]}),
          "attempts": s["attempts"]+1}

def score(s):
  chain = ChatPromptTemplate.from_template(
    "Rate 1-10 (one number only). 10=complete answer, 1=I don't know.\n{a}"
  ) | llm | parser
  q = chain.invoke({"a": s["answer"]})
  return {**s, "quality": int("".join(filter(str.isdigit,q)) or "5")}

# ── Graph ────────────────────────────────────────────────────
g = StateGraph(QAState)
g.add_node("retrieve", retrieve)
g.add_node("generate", generate)
g.add_node("score",    score)
g.set_entry_point("retrieve")
g.add_edge("retrieve", "generate")
g.add_edge("generate", "score")
g.add_conditional_edges("score",
  lambda s: "done" if s["quality"]>=6 or s["attempts"]>=3 else "retry",
  {"done": END, "retry": "retrieve"})
app = g.compile()

# ── Run ──────────────────────────────────────────────────────
out = app.invoke({"question":"What is the refund policy?",
                  "context":"","sources":[],"answer":"",
                  "quality":0,"attempts":0})
print(f"Answer: {out['answer']}")
print(f"Sources: {', '.join(out['sources'])}")
print(f"Quality: {out['quality']}/10 after {out['attempts']} attempt(s)")
